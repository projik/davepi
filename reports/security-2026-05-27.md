## SHIELD-001 — `jsonwebtoken@8.5.1` unrestricted key type

**OWASP:** A02:2021 (Cryptographic Failures), A06:2021 (Vulnerable Components)
**CWE:** CWE-327
**File:** `package.json` line 65 (`"jsonwebtoken": "^8.5.1"`)
**Caret resolves up to 8.x — does NOT pull 9.x.**

### Why this matters here
`utils/auth.js` (and login/register flows) sign HS256 tokens with `JWT_SECRET`. v8 family pre-9.0.0 allows legacy key types into asymmetric verifiers. davepi uses HS256 throughout, so direct exploit path is narrow — but **all auth in the app rides this dependency**. Stay current.

### Compliance
- SOC 2 CC6.1 (logical access — token integrity)
- PCI-DSS 6.5.10, 8.1

### Fix
```diff
- "jsonwebtoken": "^8.5.1",
+ "jsonwebtoken": "^9.0.3",
```
Then `npm i && npm test`. v9 enforces `algorithms` on `verify()` and tightens key types. Check `utils/auth.js` — if `jwt.verify` calls don't pass `{ algorithms: ['HS256'] }`, add it (v9 may error otherwise).

## SHIELD-005 — `apollo-server-core` / `-express` XS-Search bypass

**OWASP:** A05:2021
**CWE:** CWE-352 (read-only CSRF variant)

### Context
davepi pins `apollo-server-express@^3.6.3`, installs 3.13.0 wanted. Apollo v3 EOL; Apollo Server v4 is the current line (and v5 exists). Browser bug allows a malicious page to time GraphQL query responses cross-origin to infer data via XS-Search.

### Mitigation options
1. **Short-term:** ensure `csrfPrevention: true` is set on ApolloServer constructor (forces preflight on GET/multipart, blocks XS-Search vector).
2. **Long-term:** migrate to Apollo Server v4/v5. v3 is unmaintained.

### Where to look
`app.js` schema-loader block — Apollo Server instantiation. Verify `csrfPrevention: true` is in the options.

## Triage Note — Semgrep flood (1,419 hits)

The three Semgrep rule patterns in `configs/semgrep-rules/javascript.yaml` over-match:

| Rule | Why FP |
|------|--------|
| `js-nosql-injection-where` (1,340 hits) | `$WHERE` metavar = any method name → matches `arr.map(fn)`, `obj.set(x)`, anything |
| `js-prototype-pollution-merge` (73 hits) | Triggers on safe `Object.assign({}, src)` and `_.merge` even on plain objects |
| `js-xss-eval-user-input` (5 hits) | Spot-checked — all `setTimeout(controller.abort, ms)` style callbacks, not code-eval |
| `js-insecure-random` (1 hit) | One hit in `utils/migrations/lock.js:17` — used for lock-owner tag, not security |

**Spot-check verifications (real code at flagged lines):**
- `utils/webhookDispatcher.js:83,151` → `setTimeout(() => controller.abort(), timeoutMs)` — FP
- `packages/mcp/lib/http-proxy.js:71` → same pattern — FP
- `utils/schemaWatcher.js:88` → debounce `setTimeout` — FP
- `utils/migrations/lock.js:17` → `Math.random()` in lock owner ID (entropy not required) — FP
- `utils/migrations/lock.js:92` → `setInterval(renewLock, intervalMs)` — FP
- `utils/acl.js:95,106` → `out[k] = v` writes into a fresh `{}` derived from `Object.entries(body)` — FP (no `__proto__` reach into a prototype chain)

**Recommendation:** tighten the Semgrep rules upstream (Shield-claude-skill repo) — constrain `$WHERE` to the literal `where`, restrict the eval rule to literal `eval` identifier, etc. Until then, treat this rule set as advisory and audit each cluster by hand.

## Recommended Fix Order

1. **Single PR — drop-in deps bumps** (low risk, kills 7 of 9 findings):
   ```diff
   - "jsonwebtoken": "^8.5.1"
   + "jsonwebtoken": "^9.0.3"
   - "express": "^4.17.1"
   + "express": "^4.22.2"
   - "nodemon": "^2.0.13"
   + "nodemon": "^3.1.14"
   ```
   Then `npm audit fix` for `uuid` / `qs` transitives. Run `npm test`.

2. **Verify `jwt.verify` calls** pass explicit `algorithms` after v9 bump. v9 is stricter — silent regressions possible. Test login + token refresh.

3. **Enable `csrfPrevention`** on Apollo Server constructor in `app.js` schema loop. One-line change. Closes SHIELD-005 vector without a major Apollo migration.

4. **Plan Apollo v4 + Mongoose v7+ migrations** — out of scope for security PR, but both lines are EOL.

5. **Tighten or replace Semgrep rules** to cut false positive noise next scan.
