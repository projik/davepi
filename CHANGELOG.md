## [Unreleased]

### Added

- **New `schema.acl.write` document-level bypass slot: roles listed there may update records owned by another user, closing the long-standing gap where there was no way for a privileged role (e.g. an admin correcting a customer-owned record) to write across tenants.** ACL already shipped document-level `list` (read across tenants) and `delete` (remove across tenants) bypass slots plus field-level `read`/`create`/`update` gates, but update was *deliberately* hard owner-bound everywhere — the REST `PUT /:id` handler literally carried a comment noting "the spec doesn't define a write-bypass slot, so only the record's owner may PUT regardless of role," and `wrapByIdMutation` / `wrapFilter` treated `kind: 'write'` as a no-bypass kind. That left admins unable to edit a record they don't own without a hand-written custom route. The fix adds `bypassUserScopeForWrite(schema, user)` (`utils/acl.js`, mirroring the existing `bypassUserScopeForList`/`bypassUserScopeForDelete` helpers) and wires it into every update surface: REST `PUT /api/v1/<path>/:id`, the GraphQL `<path>UpdateById` / `<path>UpdateOne` / `<path>UpdateMany` resolvers *and* the state-machine transition mutations (`utils/scopeResolver.js`, where `kind: 'write'` — already the default for these resolvers — now consults the slot; `wrapStateTransition` consults it too so the validated-transition surface stays consistent with `UpdateById`, and the transition runner's `updateOne` targets the already-authorised record by `_id` alone so a cross-tenant transition doesn't silently no-op), and the `update_<path>` MCP tool (`utils/mcpServer.js`, which already honoured the `list`/`delete` bypasses for its read/delete tools). Symmetry with `acl.delete` is deliberate: just as an `acl.delete` role can `RemoveMany(filter: {})` across tenants, an `acl.write` role can `UpdateMany(filter: {})` across tenants — the slot is an explicit grant of cross-tenant reach to operator roles, so the tenancy doc's "update-everything" attack example is updated to spell out that this is intended for bypass roles (and only those). Crucially the bypass widens *which* records a role may touch, never *who owns the result*: the existing `stripTenantFields` pass at every persist site still removes `userId`/`accountId` from the update payload, so an admin edits a customer's record in place and it stays the customer's, and field-level `acl.update` still applies on top (a `write`-role caller lacking a field's `update` role still can't set that field). The REST **bulk** `PUT /api/v1/<path>` upsert is intentionally excluded and stays owner-only — its query predicate doubles as the insert seed, so dropping `userId` there could mint ownerless documents; callers wanting bulk cross-user edits use the GraphQL `UpdateMany` (which never upserts) instead. There is still no document-level `create` bypass — create always stamps ownership from the JWT, so a record is always born owned by its creator. Backwards compatible: schemas without `acl.write` are unchanged (owner-only updates), same zero-config posture as the other slots. Covered by new cases in `test/acl.test.js` (REST + GraphQL: admin edits a non-owned record, ownership preserved, stranger/hr 404s; GraphQL transition of a non-owned record by a `write` role) and `test/mcp.test.js` (`update_<path>` cross-user edit with owner preserved, stranger NOT_FOUND); docs updated across `features/acl.md`, `concepts/tenancy.md`, `reference/schema.md`, `reference/conventions.md`, the `CLAUDE.md` / `AGENTS.md` agent guides, and the `migrate-from` / `compared-to` permission-mapping tables.

## [2.1.0] - 2026-06-07

Minor: `compositeIndex` grows a long form with a per-entry `unique` flag — `{ fields: {...}, unique: false }` declares a plain compound query index (previously every entry was forcibly unique, projik/davepi-reviewer#71). Also rolls up the `davepi-plugin-oauth@0.2.0` release notes and agent-docs narrative cleanup that landed since 2.0.1.

### Added

- **`compositeIndex` entries can opt out of uniqueness: `{ fields: { userId: 1, reviewId: 1 }, unique: false }` declares a plain (non-unique) compound query index.** Until now the schema loader stamped `unique: true` on *every* `compositeIndex` entry (`utils/schemaLoader.js`), so there was no schema-level way to declare a compound lookup index at all — a consumer that wrote `compositeIndex: [{ userId: 1, reviewId: 1 }]` intending a tenant-scoped findings-by-review query index silently got a uniqueness constraint that capped the collection at **one row per (userId, reviewId) pair**, surfacing later as `E11000 duplicate key` on the second insert (found in production-shaped seeding by davepi-reviewer, projik/davepi-reviewer#71; the same rigidity previously forced that repo's partial unique index on `account.githubLogin` to be hand-created with a raw `collection.createIndex` in plugin setup). The legacy shorthand — a plain key object — is **unchanged and stays unique**: per-tenant uniqueness is the field's primary use and the shorthand has meant "unique" since it shipped, so no existing schema's behaviour moves. The long form also accepts an explicit `unique: true` (equivalent to the shorthand) and defaults to unique when the flag is omitted — `unique: false` is the only spelling that changes anything. Covered by a new `test/composite-index.test.js` (5 cases: shorthand uniqueness enforced at the database, long-form non-unique allows many rows per key pair, omitted/explicit flag defaults, both forms mixing in one schema's array). Docs: `reference/schema.md` (table row + section example), `AGENTS.md` schema-format block, and two `migrate-from/` pages whose examples used a never-valid `[{ keys }, { unique: true }]` two-entry spelling (the second entry would have been read as an index on a literal `unique` field) now show correct forms.
- **`davepi-plugin-oauth@0.2.0`: browser-SPA-friendly link flow + token delivery that keeps tokens out of URLs.** Three gaps blocked a dashboard SPA from consuming the v0.1.0 link flow cleanly. **(1) XHR-friendly link start.** `GET /auth/{provider}/link` requires a Bearer token, but a top-level browser navigation can't set an `Authorization` header, and an authed `fetch()` can't follow the cross-origin 302 to the provider (CORS-opaque) — so the link flow was only reachable from non-browser API clients. Now, when the request prefers JSON (explicit `Accept: application/json` or `X-Requested-With: XMLHttpRequest`), the route answers `200 { url }` with the authorize URL and the SPA navigates itself (`location.href = url`). Top-level navigations send `Accept: text/html,...`, so the 302 behaviour is unchanged for them. **(2) Link-mode callback honours `returnTo`.** `GET /auth/{provider}/link/callback` used to dead-end the browser on a raw JSON page. Now, when the link start carried a `returnTo` (the same validated, path-only value the login flow uses — open-redirect rules unchanged), the callback 302s back there with `?linked=<provider>` appended; the 409 already-linked-to-another-user case likewise redirects with `?error=oauth_identity_owned_by_other&provider=<id>` instead of a bare error body (and is now re-shaped as the framework's `ConflictError`, so the no-`returnTo` path surfaces as a proper 409 through `errorHandler` too). JSON fallback when no `returnTo` is unchanged. **(3) `OAUTH_SUCCESS_MODE=handler` — a login-success hook so tokens never travel in a URL.** `appendTokenToRedirect` puts the access + refresh tokens in the success-redirect query string, which transits server logs, proxies, and browser history. In `handler` mode the host app calls `registerSuccessHandler(fn)` and `fn(req, res, { tokens, user, returnTo, provider, created })` takes over the response entirely — e.g. minting a single-use handoff code and redirecting with the code in the URL fragment. Misconfiguration is fail-safe: `handler` mode with no registered handler logs an error and answers the JSON shape rather than falling back to a tokens-in-URL redirect; an unknown `OAUTH_SUCCESS_MODE` value warns and falls back to `redirect`. 11 new unit tests (68 total) cover the JSON/302 link-start split, the `linked=`/`error=` redirect composition (including `?`-vs-`&` separator handling against paths that already carry a query), handler invocation payload, the no-handler JSON fail-safe, and `registerSuccessHandler` input validation; README documents the new env var, the success-handler contract, and the SPA link-flow recipe. (#58 in davepi-reviewer's M5)

### Changed

- **Dropped the "multiple API versions" / future-`v2` narrative from the agent-facing docs so a fresh build doesn't waffle between "v1 vs v2" approaches.** README's *API Versioning* section and AGENTS.md's *Version Control* bullets advertised a future `v2/` directory at `/api/v2/`, claimed "multiple versions can coexist," and pitched "gradual migrations" — speculative framing that, with no second version actually shipping, read to an agent building on the framework as two competing ways to structure schemas and implementation. Since dAvePi has no users on a `v2` (or any migration story to preserve), that narrative is pure confusion: it has been replaced with a factual statement that schemas live in `./schema/versions/v1/` and the framework derives the `/api/v1` path prefix from the directory name. The `v1` directory and `/api/v1` prefix are unchanged — they remain the live, load-bearing path that route generation, the schema registry, the SDK, and the test suite all key off, so this is a docs/comment-only edit with no behavior change. Two stray `utils/schemaLoader.js` comments that leaned on the same framing ("the contract for v1," "cross-version relations a future caller might declare") were reworded to describe the actual scoped-then-unscoped resource lookup without implying a coming version split.

## [2.0.1] - 2026-06-02

Patch: CRM template hints + `computed` field flag in `/_describe` so the davepi-ui scaffolded admin renders relations / labels / widgets without per-resource override config out of the box. Plus a new Ollama LLM provider on `@davepi/agent` and various docs / scaffolder cleanup.

### Added

- **`computed: true` field flag passes through `/_describe`.** A field with a `computed` function on its schema definition (e.g. `fullName: { computed: (r) => [r.firstName, r.lastName].filter(Boolean).join(' ') }`) now emits `computed: true` in the manifest entry — the function itself isn't JSON-serialisable, but the flag is what consumers need. `davepi-ui` reads it and omits the field from create / edit forms (treats it like `stamped`: server-derived, client value would be ignored). Strict-equality / typeof check on the value so a stray `computed: 'true'` string doesn't slip through; computed function bodies aren't surfaced (privacy + size).
- **`@davepi/agent` gains a third LLM provider: `ollama`, for running the full agent (streaming + MCP tool-calling) against a local model with no API key.** Until now the agent supported only `anthropic` and `openai` via `LLM_PROVIDER`, both of which require a hosted key — so there was no offline / air-gapped / "develop on a flight" story, and self-hosters who already had an Ollama server running had no first-class wiring. The fix is a single new branch in `packages/davepi-agent/lib/llm/index.js#resolveModel`: `LLM_PROVIDER=ollama` reuses the already-bundled `@ai-sdk/openai` provider via `createOpenAI({ baseURL })` pointed at Ollama's OpenAI-compatible `/v1` endpoint (`OLLAMA_BASE_URL`, default `http://localhost:11434/v1`), uses `.chat(modelId)` to force `/v1/chat/completions` (the path Ollama actually implements — Ollama has no `/v1/responses`), and sets `compatibility: 'compatible'` so the SDK doesn't emit OpenAI-only request fields or strict JSON-schema tool envelopes that some Ollama models reject during tool calls. **Zero new dependencies** — `@ai-sdk/openai` is already in `package.json`, and the community `ollama-ai-provider` was rejected because of the AI-SDK-version coupling risk; reusing the first-party OpenAI provider is the same pattern Vercel themselves recommend for OpenAI-compatible servers. The orchestrator needed **no changes** because every Anthropic-specific code path already no-ops for other providers: `promptCachingEnabled()` returns `false` unless the provider is anthropic, `buildModelInput()` only injects `providerOptions.anthropic.cacheControl` when caching is on, and `emitCacheMetric()` only reads `meta.anthropic` (silently skips otherwise) — so `LLM_PROMPT_CACHING` is a documented no-op for ollama rather than a config error. `LLM_MODEL` is **required** for the ollama provider (the existing per-provider default pattern doesn't fit — Ollama users `ollama pull` their own model, so there's no universal default); an unset model throws a clear error pointing the operator at `ollama pull` and `LLM_MODEL`, instead of failing deep inside the SDK with a "model not found" once the first turn runs. `OLLAMA_API_KEY` is read but defaults to the literal string `"ollama"` because Ollama ignores the key while `@ai-sdk/openai` throws when it's missing — operators only need to set it when fronting Ollama with an auth proxy. README gains a *Local models via Ollama* section with the env table and a copy-paste quickstart, the `bin/davepi-agent.js --help` text adds the new provider row, and `package.json` adds `ollama` to keywords + the description. Covered by a new `packages/davepi-agent/test/llm.test.js` (six cases: ollama resolves with a model and no API key, missing-model throws clearly, unknown providers list ollama in the Supported set, anthropic / openai env-key validation still throws, and `DEFAULT_MODELS.ollama` is explicitly `null`).

- **Docs site grows a dedicated Agents section — ten task-shaped pages covering everything a `@davepi/agent` operator needs to create and run one, sitting alongside the existing `surfaces/agent.md` reference (which is now flagged as the reference and cross-links into the new section).** The reference page has been a single 290-line dump since #128 and is increasingly the only place new agent surfaces (persona/memory, the frozen snapshot, skill governance, proactive cron, Ollama) get a paragraph — fine as a lookup, not enough to actually onboard someone. The new section under `docs/site/src/content/docs/agents/` is organised by task: `index.md` (mental model + ACL boundary + when-to-use matrix), `getting-started.md` (zero-to-working-chat in ~5 min against a local davepi), `configuration.md` (full env + file-config + programmatic-override reference with precedence rules and three worked end-to-end configs — storefront widget, per-user multi-tenant support bot, air-gapped Ollama dev), `personas-memory-skills.md` (the whole learning layer with the prompt-slot diagram, frozen-snapshot discipline, multi-agent-per-tenant model, schema field tables, and the agent service-role contract), `auth.md` (service vs per-user with the link-flow sequence diagram and the *why-not-URL-tokens* rationale from PR #128 review #8), `channels.md` (HTTP + Slack first-class, plus a worked-example *build a Discord channel* recipe pointed at the template stubs), `tools-and-rendering.md` (MCP auto-derivation + the tool router meta-tools + render-tool safety + a *typical turn vs routed turn* sketch), `proactive-agents.md` (cron + approved skills + cancellation semantics + per-user-cron registration error), `programmatic-api.md` (`createAgent` / `runTurn` / `mcpClient` for embedding the agent in another process, with a worked dispatcher pattern for multi-tenant single-process deployments), and `troubleshooting.md` (a triaged checklist: boot failures, auth errors, stuck link flows, missing tools, Slack 4xx, Anthropic cache misses, Ollama tool-call quirks, "it works on my machine"). Each page links into the others and back out to the relevant feature/concept pages so the IA stays a graph, not a list. `docs/site/astro.config.mjs` gets the new sidebar entry between *Tutorial series* and *Concepts* — placement chosen so a user who finishes the tutorials lands on the deeper agent docs before being asked to grok the framework concepts under them, and so *Surfaces → Agent* (the reference) lives downstream as one would expect. `surfaces/agent.md` gains a starlight `:::tip` callout at the top pointing readers at the new section. No code changed — this is doc-only, so the [skip-changelog](https://github.com/projik/davepi/blob/main/.github/workflows/changelog.yml) gate doesn't strictly need an entry, but the section is large enough that a release note is warranted.

### Changed

- **CRM template schemas adopt the M0.5 hint surface.** `account` / `contact` / `deal` / `activity` all declare schema-level `label` / `pluralLabel` / `displayField` (so the davepi-ui sidebar reads "Accounts" not "account", and relation pickers / breadcrumbs surface a useful preview field instead of `_id`). Field-level: `userId` marked `stamped: true` on every template (the JWT-stamped tenant field disappears from forms), `parentAccountId` / `contactId` / `dealId` gain `reference: 'account' | 'contact' | 'deal'` so the admin's `<RelationPicker>` resolves against the right collection instead of demanding a UUID paste, `website` gets `widget: 'url'`, `description` / `body` get `widget: 'textarea'`, `email` gets `widget: 'email'`, `deal.amount` gets `widget: 'currency'` + `format: 'currency:USD'`, `activity.type` gains an explicit `enum: ['call','email','meeting','note']` so the form renders a Select, every camelCase / abbreviated field gains an explicit `label` (`parentAccountId` → "Account", `firstName` → "First name", `expectedCloseAt` → "Expected close"). The `account.primaryContact` `hasOne` relation stays declared for programmatic callers but the admin UI now suppresses its tab when the parent already has an equivalent `hasMany` against the same FK (the `Contacts` tab covers the same data — `Primary Contact` next to it was UX clutter — see davepi-ui 0.2.x).

- **Docs walk users through the new davepi-ui admin flow instead of the removed `/admin` SPA.** `quickstart/`, `guides/crm-in-10-minutes/`, `tutorials/habit-tracker/`, and `tutorials/customer-support-inbox/` all swap their "open `http://localhost:<port>/admin`" steps for "boot the sibling `<project>/admin/` davepi-ui app at `http://localhost:5173` via `cd admin && npm install && npm run dev`". The landing-page feature card flips from "Refine + ant-design" to "davepi-ui: shadcn + Tailwind. Searchable relation pickers, auto-discovered child tabs, scaffolded by default with `create-davepi-app`." `compared-to/refine.md` gets a full rewrite reframing davepi-ui as the Refine alternative dAvePi ships out of the box — the "use both" sketch still applies for teams who want a custom Refine admin pointed at the davepi backend (`create-davepi-app --no-admin` then start a fresh Refine project against `davepi gen-client`'s output). The `create-davepi-app` next-steps print + scaffolded README likewise drop their `pnpm install` / `pnpm dev` lines for `npm install` / `npm run dev` so the user's package-manager surface stays aligned with the davepi backend (the scaffolded admin is a plain Vite app — davepi-ui itself keeps pnpm internally because it's a monorepo, but the consumer-facing template ships without lock files so the scaffolded project picks npm fresh).

## [2.0.0] - 2026-06-02

Major: bundled Refine admin removed. Replaced by [davepi-ui](https://github.com/projik/davepi-ui), shipped as separate npm packages (`@davepi/ui-core`, `@davepi/ui-react`, `@davepi/ui-mcp`, `create-davepi-ui`). `npx create-davepi-app` scaffolds a davepi-ui sibling admin by default — opt out with `--no-admin`.

### Removed

- **Deleted the bundled Refine admin SPA at `/admin`.** Replaced by [davepi-ui](https://github.com/projik/davepi-ui), a separate schema-driven, agent-first React admin built on shadcn primitives — searchable relation pickers (not raw UUID inputs), auto-discovered child tabs on parent detail pages, per-resource override layer, JSON page descriptors, MCP server. The old Refine admin was bundled inside the davepi npm package and served from `/admin`; the new admin runs as a sibling Vite project pointed at the davepi backend via `VITE_API_URL`. davepi has no users today so no deprecation window — the dir / route / helmet carve-out / nodemon-ignore entries / jest path filter / npm-publish artifact / scaffolder note / docs reference all go in one cut. Files: removed `admin/` directory, the helmet `/admin/*` carve-out and the `/admin` express.static + catch-all in `app.js`, the `install:admin` / `build:admin` / `dev:admin` / `prepublishOnly` npm scripts in `package.json`, the `admin/dist/` entry from `files`, the `admin/*` patterns from `nodemonConfig.ignore` and `jest.testPathIgnorePatterns`, the `test/admin-ui.test.js` integration test, the `/admin/*` CSP carve-out assertion in `test/security.test.js`, the `Admin SPA | admin/` row in `MAINTAINERS.md`, and the `admin/` line in `CONTRIBUTING.md`'s directory map. README's Admin UI section now points directly at davepi-ui's scaffolder (`npx create-davepi-ui` or `npx create-davepi-app --template crm`, which scaffolds the admin by default — opt out with `--no-admin`).

### Changed

- **`create-davepi-app` scaffolds a davepi-ui sibling admin by default.** A new project gets both the davepi backend at `<name>/` and a freshly-scaffolded davepi-ui Vite app at `<name>/admin/` pre-pointed at `VITE_API_URL=http://localhost:<port>`. Opt out with `--no-admin` (e.g. for headless API-only deployments). The admin scaffold is delegated to `npx -y create-davepi-ui` at run time so the scaffolder logic stays in the davepi-ui repo and tracks whatever the latest published version produces — no template duplication on the davepi side. `--no-install` continues to skip the backend's `npm install`; the admin sub-project's `pnpm install` is deferred regardless (kept off the critical path so a slow registry there doesn't block the rest of setup). The scaffolded project's README + next-steps print now include `cd admin && pnpm dev` (port 5173) alongside `npm start` (port 4001).

### Added

- **Seed schemas adopt the M0.5 hint surface end-to-end.** Now that `/_describe` carries `label` / `pluralLabel` / `displayField` (schema) and `label` / `widget` / `format` / `enum` / `stamped` (field) plus auto-populated inverse `hasMany` relations, every seed schema declares these hints so a downstream admin (davepi-ui in particular) renders the right things by default without per-resource override config. Schema-level: `label` / `pluralLabel` / `displayField` on all six CRM resources (`account`, `category`, `contact`, `product`, `project`, `quote`). Field-level: `userId` and `accountId` on every seed schema (CRM + `agentMemory` / `agentPersona` / `apiClient` / `conversation` / `customerProfile` / `skill`) carry `stamped: true` so the admin UI hides them from create / edit forms — they're populated from the JWT by the auto-generated handlers and a client-supplied value is ignored, so showing them as inputs was always a doomed-override trap. `contact.email` gains `widget: 'email'`, `product.price` gains `widget: 'currency'` + `format: 'currency:USD'`, address / name fields gain explicit `label` overrides (`first_name` → "First name", `sku` → "SKU"). `searchable: true` lands on every field a user would reasonably full-text against (`name` / `description` on CRM resources, `first_name` / `last_name` / `company` on contact) — `accountName` excepted intentionally to preserve the canonical "schema with no searchable fields" test fixture (`test/full-text-search.test.js`, `test/public-read.test.js` all rely on `account` for the negative case). Relations: `quote` declares `belongsTo: contact` (the only real cross-resource parent FK in the seed; `accountId` everywhere else is the tenant marker, hence stamped), so `buildManifest()` auto-populates the inverse `quotes` hasMany on `contact` and a contact's detail page can render a Quotes tab without manual schema authorship. `category` and `project` self-references (`parent` field) become explicit `belongsTo` declarations so the RelationPicker resolves the parent against the same collection. One new live test in `test/describe.test.js` asserts the full hint set on the seed `/_describe` response (display hints, stamped flags, widget / format / label passthrough, the synthesised inverse from `quote.belongsTo: contact`, and the self-reference declarations). The existing `mcp.test.js` "search_<path> not registered" case swaps `product` for `account` to match the new searchable surface. 25/25 describe tests + 25/25 mcp tests + 22/22 public-read tests in isolation all green; full suite shows the same two pre-existing flake suites (`idempotency`, `public-read` under pollution) that already failed on `main` before this change.

- **Surface frontend display hints + auto-populate inverse `hasMany` relations in `/_describe`, so an agent-first admin (or any consumer of the manifest) doesn't have to sniff for them client-side.** Schema-level: `label` / `pluralLabel` / `displayField` now pass through to the manifest entry so a `<RelationPicker>` over `account` reliably shows `accountName` (or whatever the schema declares) instead of falling back to "first searchable String field" heuristics. Field-level: `label` / `widget` / `format` / `enum` / `stamped` travel through to the per-field entry. `widget` names a widget kind (`rich-text`, `textarea`, `email`, `url`, `currency`, …) the consumer dispatches on; `format` carries a value-formatting hint (`currency:USD`, `date`); `enum` is the long-overdue first-class surfacing of mongoose's `enum` array (consumers were re-deriving it from the model otherwise); `stamped: true` declares a field the server fills in from the JWT (`userId` / `accountId` on the seed schemas, but consumers can mark their own) so a frontend can hide stamped fields from create / edit forms entirely instead of letting users override values the backend will replace. Pure metadata, ignored by the framework. `buildManifest()` now also auto-populates the inverse `hasMany` edges on every parent: for each `belongsTo` declared on schema B pointing at A, the manifest emits `A.relations.<plural B> = { kind: 'hasMany', target: B.path, foreignKey: <localKey>, inverse: true }`, unless A already declares a relation against the same target + foreign key (consumer-authored relations always win — the `inverse: true` discriminator lets the GraphQL builder distinguish synthetic edges if it ever wants to). Inverse population is idempotent and silently drops `belongsTo` declarations whose target isn't registered. Together with the schema/field hints, the [davepi-ui](https://github.com/projik/davepi-ui) `SchemaRegistry` can drop its FK-by-convention sniffing and rely on the manifest as the source of truth — covered by eight new pure-helper tests in `test/describe.test.js`.

- **Completed per-package npm release automation so every publishable package in the workspace ships through the same tag-triggered, reviewer-gated pipeline.** Two packages had no publish workflow at all — `@davepi/agent` and `davepi-plugin-skill-extractor` — so the only way to release them was a manual `npm publish`, bypassing the version-vs-tag check and the `npm-publish` environment approval gate every other package goes through. Added `.github/workflows/davepi-agent-publish.yml` and `.github/workflows/davepi-plugin-skill-extractor-publish.yml`. The skill-extractor workflow mirrors the slack/postmark template verbatim (tag `davepi-plugin-skill-extractor@*`, no `npm install` because the entrypoint and its `node:test` suite only *lazy*-require the heavy peers `ai` / `@ai-sdk/anthropic` / `davepi-plugin-queue` from inside functions, `npm test`, then `npm publish --access public` with `NPM_TOKEN` gated behind the `npm-publish` GitHub Environment). The agent workflow deliberately diverges from the "run the package's `npm test`" template: `@davepi/agent`'s suite is a monorepo *integration* suite that requires `davepi/app.js`, `mongodb-memory-server`, and `supertest` — none declared as the package's own deps — so an isolated `npm test` in `packages/davepi-agent` would fail to resolve them. Instead it mirrors the `@davepi/mcp` packaging-smoke posture: a `node bin/davepi-agent.js --help` check (the bin handles `--help` and exits 0 *before* requiring any heavy dep) plus an `npm pack --dry-run` allowlist asserting `index.js` / `bin/davepi-agent.js` / `lib/*` / `README.md` ship, then `npm publish --access public`. - **API keys for programmatic access (`POST` / `GET` / `DELETE /api/auth/api-keys`).** Sessions until now meant a short-lived JWT plus a refresh-token rotation chain — fine for a human in a browser, useless for a CI job or a server-to-server caller that needs a long-lived, revocable credential it can paste into a secret store. This adds first-class API keys minted per user. A new `model/apiKey.js` stores only the SHA-256 hash of the secret (never the plaintext, which is shown exactly once at creation), plus a display `prefix` (`dpk_xxxx`), the minting user's `roles` and `email` **frozen at creation** (read back from the key row on every request, never re-fetched from the User — so a key can never be elevated past what its owner held when it was minted), a coarse `scopes` array (`['read', 'write']` default), and nullable `expiresAt` / `revokedAt`. The auth surface resolves them through one shared helper, `utils/tokens.js#resolveApiKeyUser`: any bearer value starting with `dpk_` (instead of being parsed as a JWT) is hashed and atomically looked up via `findOneAndUpdate` — the predicate rejects revoked/expired rows and the same write bumps `lastUsedAt` in one round-trip — then populates the *same* `req.user` shape a JWT produces (`user_id`, `email`, `roles`) plus `scopes` and `authMethod: 'apiKey'`, so every existing tenant-scoping, ACL, and GraphQL `scopeResolver` path keeps working unchanged. Both `middleware/auth.js` (REST) and `app.js`'s `buildGraphqlContext` (GraphQL) call the same helper, so a key authenticates identically on both transports; a miss throws `UnauthorizedError`, the same class JWT failures use, so `errorHandler` shapes it identically (401). Coarse scopes are enforced by a new `requireScope('read' | 'write')` — a tiny `utils/scopes.js#hasScope` (a request with no `scopes` array, i.e. a JWT or `X-Client-Id` session, implicitly holds every scope, so behaviour for non-key callers is byte-for-byte unchanged) wrapped as `middleware/requireScope.js` for REST (mounted after `auth(true)` on every generated route: `GET` → `read`, `POST`/`PUT`/`DELETE` → `write`) and inlined into the `scopeResolver` wrappers for GraphQL (`wrapCreateOne`/`wrapCreateMany`/`wrapByIdMutation`/`wrapStateTransition` and write-class `wrapFilter` gate on `write`; the read wrappers and `wrapAggregation` gate on `read`). Minting requires a JWT — an API-key-authenticated request that POSTs to `/api/auth/api-keys` is refused with 403, so a key can never bootstrap another key — while listing returns the caller's own keys with `tokenHash` projected out and the plaintext gone forever, and `DELETE` flips `revokedAt` (tenant-scoped: revoking another user's key is a 404). Backwards compatible — no existing route signature changes and no new env vars. Covered by `test/apiKeys.test.js`.

- **`@davepi/agent` learning layer, workstream D: skill extraction from successful conversations (`docs/agent-learning-layer.md` §7).** Workstream C made skills governed records a human curates, but every runbook still had to be hand-authored; the agent never turned a good outcome into reusable knowledge the way Hermes's self-improvement loop does. This slice closes that loop while preserving C's safeguard that nothing reaches a live customer unreviewed, and it does so by composing primitives already in the tree rather than adding a privileged path. **(1) Resolution is a state transition that fires a transcript-bearing event.** `schema/versions/v1/conversation.js` gains a `status` state machine — `open → resolved` (a good outcome) or `open → abandoned` (dropped); both terminal, so a closed thread isn't reopened in place. There's no field ACL on `status`, so the agent's service role closes its own conversations through the normal update surface (identical on REST, GraphQL, **and** MCP). Arriving at `resolved` runs an `onEnter` hook that emits a dedicated `conversation.resolved` record event on `utils/events.js` carrying the **full transcript** — we emit our own event rather than lean on the generic `${path}.transitioned` one because the MCP/GraphQL transition events deliberately omit the `record` payload, so a subscriber there would have no `history` to extract from, and because it's the exact semantic signal the loop subscribes to. `userId` rides as the tenant owner so any proposed skill scopes back to the originating account. **(2) New package `davepi-plugin-skill-extractor` runs extraction off-thread.** It subscribes to `conversation.resolved` and hands the work to `davepi-plugin-queue` (extraction is slow + pure-upside, so it's best-effort and asynchronous — resolving a conversation never blocks the response). The queued job carries only identifiers + tenancy (`userId`, `accountId`, `agentKey`, `recordId`), never the transcript itself — `history` is the full JSON re-serialized every turn and grows without bound, so the worker re-reads it from the conversation record by id rather than duplicating an ever-growing blob into Redis on every resolution (a record deleted before the job runs is logged and skipped). The queue worker runs a **fresh** extraction agent (`lib/agent.js`, mirroring the "cron job = fresh instance" discipline and `eval/lib/agent.js`): a single, *tool-free* LLM call that reads only the transcript — it can take no action on tenant data — and replies with strict JSON, proposing a skill **only** when the approach was non-trivial AND the outcome positive (`{ "skill": null }` otherwise, the common case). A cheap pre-filter skips transcripts shorter than a couple of exchanges before any LLM call, and the verdict parser tolerates code fences / surrounding prose. **(3) The proposal lands as a governed draft, tenant-scoped.** `lib/persist.js` writes the skill via the schema-generated `skill` model, stamping `userId`/`accountId` from the originating conversation (never the worker's ambient identity) and forcing `status: 'draft'` itself — the direct model write bypasses `stampInitialStates`, so the worker re-asserts the governance contract, keeping the extracted runbook invisible to the L0 prompt index until an operator promotes it via the #131 state machine. Skill names are unique per `(account, agentKey)`, so the persist step looks up by name and skips rather than duplicating or clobbering an existing skill — re-resolving a conversation (or a similar one) never resurfaces or overwrites a skill an operator may have already approved or edited. The LLM call, queue, and skill model are all injectable, so the plugin is dormant-safe (no Redis → logs a warning and ignores events, never an error) and fully unit-testable without Redis/Mongo/an API key. Acceptance criteria — a non-trivial resolved conversation producing a coherent draft skill without blocking the response, trivial chats producing none, and the proposal tenant-scoped to the originating account — are covered end-to-end against the real schema layer + event bus + skill model in `test/agent-learning-loop.test.js`, plus the package's own `test/extract.test.js` and `test/plugin.test.js`.
- **`@davepi/agent` learning layer, workstream C: skills as governed records with L0/L1/L2 progressive disclosure (`docs/agent-learning-layer.md` §6).** Workstreams A/B gave the agent an identity and slow-changing memory, but no *procedural* memory — reusable runbooks ("how we issue a refund") it can retrieve and follow — and no governance to keep a self-authored runbook from reaching a live customer unreviewed. This slice adds both, reusing machinery the framework already has rather than inventing a new subsystem. **(1) New `schema/versions/v1/skill.js`** — fields `accountId`, `userId`, `agentKey`, `name` (indexed, `searchable`) + `description` (`searchable`) as the cheap L0 teaser, `body` (markdown) as the on-demand L1 detail, `attachments` (object-storage `File`, `private` access) as L2, a `useCount` counter for the learning loop (#132) to rank promotion/retirement, and a `status` field carrying a `stateMachine` (`draft → approved → deprecated`). Skill names are unique per `(userId, agentKey, name)` so operators and the learning loop can address a skill by name. It declares no `acl.list`/`acl.scope` bypass, so the per-`userId` owner scope gives tenant isolation for free across REST, GraphQL, **and** MCP. **(2) Progressive disclosure in `packages/davepi-agent/lib/promptAssembly.js`** — a new `renderSkills` produces prompt **slot #3** (between the operating contract and memory): the L0 index lists each approved skill's `name`, `description`, and `_id`, and instructs the model to call the schema-generated `get_skill` tool to read the full `body` (L1) before acting — the same "load detail only when selected" discipline the tool router applies to tool schemas, now applied to knowledge. The index is capped (50 lines, tight per-line limits) so a tenant with hundreds of skills can't flood the cache-stable prefix, and `name`/`description` pass through the same prompt-injection sanitizer as persona/profile text. `orchestrator.js` gains `makeSkillsFetcher`, which lists `status: 'approved'` skills for the agent's `agentKey` via `list_skill` (ordered by `useCount`) and freezes them into the session snapshot like every other slot; a missing schema, empty result, or thrown fetch simply omits the slot, keeping the zero-config path byte-identical. L2 attachments are pulled through the existing object-storage file tools only when the body references them. **(3) Governance via state machine + field ACL, enforced on every write surface.** `stampInitialStates` forces `status` to `draft` on every create (REST, GraphQL, MCP) regardless of what the caller supplies — re-asserted by a `beforeCreate` hook as defence-in-depth — so an agent (or a forged `{ status: 'approved' }`) can never author a live skill. The `status` field declares `acl.{create,update}: ['user','admin']`, so `filterWritable` strips it from any write by the `agent` service role on every surface, meaning only a human operator can transition `draft → approved`; the state machine then constrains the operator to the one-way `draft → approved → deprecated` lifecycle, where `deprecated` is **terminal** — a retired runbook is never re-approved in place (which would silently re-enter the L0 index), so authoring a fresh skill is the path back. Because the L0 index only ever lists `approved` skills, a half-baked self-authored runbook stays invisible to customers until a human signs off, and a `deprecated` skill drops out of the index for good. A `beforeDelete` hook also refuses agent-authored deletes (delete has no field-level ACL and the agent shares the owner's `userId` in service mode); it runs on every by-id delete surface including MCP `delete_skill` (see the framework fix below), so an agent can't route around the gate by calling the MCP tool directly. Acceptance criteria — an approved skill's L0 appearing in the prompt with the model fetching L1 first, an agent-created skill landing as `draft` and invisible until approved, deprecated skills disappearing, and tenant isolation across REST/GraphQL/MCP — are each covered by tests (`test/agent-skill.test.js`, `packages/davepi-agent/test/promptAssembly.test.js`, `packages/davepi-agent/test/orchestrator.test.js`).

- **`@davepi/agent` learning layer, workstream A: agent persona (the Hermes `SOUL.md` analog).** First slice of the agent "learning layer" RFC (`docs/agent-learning-layer.md` §4): each agent now has an owned, editable identity that leads its system prompt. The motivation is that PR #128 shipped the agent *body* (orchestration, channels, MCP-derived tools) but hard-coded one `DEFAULT_SYSTEM_PROMPT`, so a tenant running several agents (support, sales, ops) had no per-agent brand voice and no seam for the memory/skills tickets (B/C) to plug into. The fix lands as a new schema-driven resource plus a prompt-assembly seam, deliberately keeping everything tenant-isolated server-side rather than as flat files on the agent host. **(1) New `schema/versions/v1/agentPersona.js`** — fields `accountId`, `userId` (owner/operator), `agentKey` (unique per account via a `(userId, agentKey)` composite index, so one tenant hosts many agents), the four Hermes sections `identity` / `style` / `avoid` / `defaults`, a `status` enum (`active`/`archived`), and a server-written `proposedPatch`. It declares no `acl.list`/`acl.scope` bypass, so the default per-`userId` owner scope every auto-generated route enforces gives tenant isolation for free — account A's credentials cannot read account B's persona over REST, GraphQL, **or** MCP (all three covered by tests). The agent reads its persona under its own MCP identity, which in service mode owns the tenant's data; nothing here opens a cross-tenant read path. **(2) New `packages/davepi-agent/lib/promptAssembly.js`** — `orchestrator.js` now calls `assembleSystemPrompt()` instead of using the hard-coded constant. A persona renders as prompt **slot #1** ahead of the operating contract (slot #2); with no persona row, no `agentKey` configured, an archived/empty persona, or a backend that lacks the schema, it falls back to exactly the previous `DEFAULT_SYSTEM_PROMPT` so the zero-config path is byte-identical to before. A new `agent.key` config field (env `AGENT_KEY`) selects which persona this process reads. **(3) Sanitizer over persona text before it enters the prompt** — `sanitizePersonaText` strips C0 control characters, neutralises prompt-injection phrasing (`ignore previous instructions`, `new instructions:`) and forged role turns (`system:` line-starts, `<system>…</system>` tags), caps each section at 2000 chars, and logs once when it trips. This belongs in the seam from day one because later tickets feed end-user-derived text (customer profiles) through the same assembly path, making it an injection vector into a future session's identity tier. **(4) Operators own the persona; agents are write-gated on every surface.** Operators own brand voice and compliance "avoid" rules, so an `agent`-role caller must not be able to author, rewrite, or delete a live persona — and that has to hold on MCP and bulk paths, which do **not** run schema lifecycle hooks. So the gate is the two layers that *are* universal (the same posture `apiClient` uses): field-level ACL (`{ create/update: ['user','admin'] }`) on every live field — `agentKey`, `identity`, `style`, `avoid`, `defaults`, `status` — so `filterWritable` strips them from any agent-authored create/update across REST single + bulk PUT, all GraphQL create/update mutations, and MCP create/update (and because `agentKey` is `required`, an agent-authored create fails outright); plus a `beforeDelete` hook refusing agent-authored deletes on the by-id paths (delete has no field-level ACL and the agent shares the owner's `userId` in service mode). The agent's only writable field is `proposedPatch`: it proposes edits by writing that field, and an operator reviews and applies them to the live sections. (This replaces an earlier `beforeUpdate` hook that rerouted agent edits into `proposedPatch` — a hook can't enforce that on MCP/bulk, and it clobbered a pending patch on any agent update that carried no section fields.) The deployment contract is that the agent's service token carries role `['agent']` (not `user`/`admin`) while sharing the tenant `userId`, so it reads the persona it owns but cannot mutate it. Acceptance criteria — two agents on one tenant rendering distinguishably different voices from their persona rows alone, persona text sanitized before injection, the missing-persona fallback, and tenant isolation across REST/GraphQL/MCP — are each covered by tests (`test/agent-persona.test.js`, `packages/davepi-agent/test/promptAssembly.test.js`).
- **`@davepi/agent@0.1.0` published.** New first-party package under `packages/davepi-agent/`, sibling to `@davepi/mcp` but solving the opposite half of the agent story: where `@davepi/mcp` is a stdio↔HTTP byte pump that lets developer tools (Claude Desktop / Cursor / Code) connect to a dAvePi instance for editing-time use, `@davepi/agent` is an **end-user-facing chat agent** that ships pre-wired against a running dAvePi backend so any consumer project can stand up a customer-service / internal-tools chatbot with one `npx davepi-agent` invocation. The motivating gap: every davepi consumer who wanted a chatbot had to glue together an LLM SDK, an MCP client, channel adapters, an auth refresh loop, and chart/table rendering themselves — work that's substantially the same project to project because the davepi `/mcp` endpoint already exposes every schema's CRUD + relations + aggregations + audit + file ops as tools with tenant isolation, role-based ACL, and role-scoped mandatory filters enforced server-side. The agent collapses that integration to a single package. (1) **MCP client (`lib/mcpClient.js`).** Uses `@modelcontextprotocol/sdk`'s `StreamableHTTPClientTransport` against the davepi `/mcp` HTTP endpoint, NOT inlined into the davepi process the way a plugin would be — the agent runs as a separate process and authenticates as a client, which is the right posture for "an agent that acts on behalf of a specific user / role" and matches what an embedded widget or Slack bot would do. Auth headers are attached **per request**, not at client construction, so per-user deployments can rotate identity between turns without rebuilding the transport. Tool list is cached and re-pulled lazily; the SDK is dynamically imported so the package can target CommonJS while the SDK ships ESM-only modules. (2) **Two auth strategies, selectable per deployment via `AGENT_AUTH_MODE`** (`service` default, `per-user` opt-in). Service mode (`lib/auth/service.js`) reads `DAVEPI_BEARER` or `DAVEPI_CLIENT_ID` from env once — Bearer wins when both are configured, mirroring `middleware/clientAuth.js`'s posture; X-Client-Id is the right answer for an anonymous storefront widget paired with an `apiClient` row whose role has `schema.acl.scope[role]` filters declared, and davepi will refuse writes from `isClient: true` callers automatically. Per-user mode (`lib/auth/perUser.js`) stores davepi refresh tokens keyed on `(channel, channel_user_id)` in a pluggable store (default SQLite via `better-sqlite3`, `memory:` for ephemeral / test use, `STORE_URL` override) and mints fresh access tokens at `refreshSkewSeconds` (default 60s) before expiry — caching the access token so a typical turn doesn't burn a refresh round-trip. First contact from an unlinked user throws `UNLINKED` with a `linkUrl` that points at davepi's `/login?redirect_uri=<agent>/oauth/callback?nonce=<one-time>`; the agent's HTTP channel mounts the `/oauth/callback` to complete the link. The orchestrator catches `UNLINKED` and surfaces the link URL as the assistant message rather than letting it raise as a 5xx — the right UX for "you need to log in before I can look up your data." The design rule throughout: **the JWT (or X-Client-Id) IS the access boundary; the agent never re-implements ACL checks and never tries to constrain results via prompt text** ("only show user X's orders") — that's a confused-deputy bug waiting to happen, and the README states it explicitly. (3) **Orchestrator (`lib/orchestrator.js`) on the Vercel AI SDK.** One provider switch (`lib/llm/index.js`) keyed off `LLM_PROVIDER` covers Anthropic (default, `@ai-sdk/anthropic`) and OpenAI (`@ai-sdk/openai`); new providers slot in by adding one factory. MCP tool input schemas (JSON Schema, as returned by `tools/list`) flow through the AI SDK's `jsonSchema()` helper into tool definitions — no JSON-Schema-to-Zod converter shipped in the package, which would have been a tax with no payoff given the helper exists. Streaming is on by default; `streamText`'s `onStepFinish` relays tool calls and results to the channel adapter via an `onEvent` callback so each channel can decide how to render them. The system prompt explicitly tells the model NOT to add "for user X" filters and to trust empty-result tool returns instead of widening the search. (4) **Tool router (`lib/toolRouter.js`)** kicks in when the live tool count exceeds `AGENT_TOOL_LIMIT` (default 40). Most LLMs degrade past ~50 tools — too many slots in the system message, weaker tool-pick precision. Routed mode exposes only two meta-tools (`list_resources`, `use_resource(name)`) plus the render tools; once the model picks a resource, the orchestrator narrows the exposed tool list to tools whose name contains that resource for the next turn. Resource bucketing parses davepi's tool naming convention (`list_<resource>`, `get_<resource>`, `create_<resource>`, etc.) — pure client-side, no davepi change. (5) **Render tools (`lib/renderTools.js`).** Two client-synthesized tools (`render_table`, `render_chart`) the model can call when it has tabular or chartable data. Inputs are Zod-validated (columns + rows for tables; Vega-Lite v5 spec for charts) so a prompt-injected response can't smuggle raw HTML/SVG into the channel. Render events are forwarded through the orchestrator's `onEvent` stream; each channel adapter renders natively. (6) **HTTP channel (`lib/channels/http.js`).** `POST /chat` with SSE streaming (`token` / `tool_call` / `tool_result` / `render` / `final` event types) plus a non-streaming JSON envelope mode for callers without an SSE client. `GET /health` for liveness. `GET /oauth/callback` mounted only in per-user mode. CORS allowlist via `AGENT_CORS_ORIGINS`. Errors after SSE headers are surfaced as `error` events on the stream rather than HTTP status codes, since the response is mid-flight. (7) **Slack channel (`lib/channels/slack.js`) on `@slack/bolt`.** App-mention in a channel → reply in thread; DM → reply in DM. Per-conversation history kept in-memory keyed on `(channel, thread_ts)`; intentionally not persisted, restart resets — operators who need persistence can wire a store later. Render events translate channel-natively: `render_table` becomes a Block Kit `section` with a mrkdwn table (fenced code-block fallback for >10 columns), `render_chart` becomes a QuickChart image URL embedded as an image block (Vega-Lite spec → URL-encoded into the QuickChart query string, no SVG rendering inside Slack). Both HTTP and socket-mode supported via `SLACK_SOCKET_MODE`. UNLINKED errors during a Slack turn fall back to posting the link URL in-thread. (8) **Channel templates** (`lib/channels/templates/{telegram,whatsapp,widget}.js`) are checked-in stubs with the recipe for filling them in — they mirror the `http.js` / `slack.js` shape (`channelCtx` derivation, `runTurn` invocation, render-event translation) so a v2 contributor adds one file plus the relevant SDK dep without re-deriving the architecture. The embeddable widget specifically is a browser bundle, not a server-side channel — it talks to the existing `/chat` SSE endpoint. (9) **`createAgent()` / `startAgent()` programmatic API** in `index.js` for consumers who want to embed the orchestrator in their own process rather than running the `davepi-agent` bin — the same `{ config, auth, mcpClient, model }` quad the bin uses is returned, and `runTurn()` is exported so a custom channel can drive the orchestrator directly. - **`davepi-plugin-cron@0.1.0` published.** Sixth first-party plugin under `packages/davepi-plugin-cron/`, modelled on the slack / postmark / audit / object-storage / queue packages and covering the universal "nightly export at 2am", "reap stale uploads every 10 minutes", "send digest emails Monday 8am" need that every project eventually bolts on. The motivating gap: a naive `node-cron` install scales to one process, fine; scale to two web dynos and every job runs twice. Solving that properly — distributed lock, leader election, lease expiry, heartbeat for long runs, cooperative abort — is enough work that it belongs as one of the standard plugins. The package wraps [croner](https://github.com/Hexagon/croner) (zero-dep, timezone-aware scheduler) and adds a `cron_lock` collection with a unique index on `name` plus a TTL index on `expiresAt` so exactly one node per tick wins the upsert race and a crashed leaseholder's row gets swept by Mongo. The decision to NOT ride on `davepi-plugin-queue`'s BullMQ `repeat` (which #114 already exposes) is that this plugin uses the existing Mongo connection — zero new infra, no Redis required — and is the right answer for projects that haven't added Redis yet. Both can coexist; the README's comparison table picks a line between them. (1) **Declarative `davepi.cron` block in `package.json`** mapping `name → { schedule, handler, timezone?, leaseSeconds? }` is the primary surface. The handler spec resolves via the same posture `utils/pluginLoader.js#resolvePlugin` uses (relative paths against cwd, absolute paths as-is, package names through Node's resolution starting at cwd) and the resolved module is a function (or `{ handler, ... }` with overrides). A typo'd declaration throws at boot — a silently-disabled job is harder to diagnose than one that fails loudly. (2) **Programmatic `register(name, opts)` / `unregister(name)`** for the dynamic-list case (e.g. one job per active tenant, hot-reload during dev). `register` after `setup()` schedules immediately; double-registration of the same name throws so a misconfigured `davepi.cron` with duplicate keys doesn't silently drop one. (3) **Mongo-backed distributed lock (`lib/lock.js`).** Every tick runs `findOneAndUpdate({ name, expiresAt: { $lt: now } }, { $set: { holderId, expiresAt: now + lease } }, { upsert: true })` — the unique index on `name` makes the upsert race-safe, exactly one of N contenders' upserts succeeds, the others throw `E11000` which the plugin maps to "another holder owns it; skip this tick." The lock — not the scheduler — is the source of truth, so clock skew between dynos that causes the same cron expression to fire on two nodes within the same second is caught at the lock. A `expireAfterSeconds: 0` TTL index on `expiresAt` lets Mongo sweep rows whose owners crashed mid-run. (4) **Heartbeat + AbortSignal for long-running jobs.** While the handler runs, a background `setInterval` ticks every `leaseSeconds/3` and extends `expiresAt`. If a heartbeat returns no document (the row's `holderId` no longer matches ours because the lease elapsed and someone else took over), the lease's `AbortSignal` flips so the handler can stop cooperatively (`if (signal.aborted) return`). The timer is `unref`'d so a CLI / test that exits before the next heartbeat doesn't wait. (5) **Per-job timezone** flows from `davepi.cron[name].timezone` (or the programmatic equivalent) through to croner's `timezone` option; `CRON_DEFAULT_TZ` (default `UTC`) is the fallback for jobs that don't specify one. (6) **Admin-gated status route** `GET /api/cron` (configurable via `CRON_STATUS_PATH`, empty disables) returns the list of registered jobs with `nextRun` / `lastRun` / `lastStatus` / `lastDurationMs` / `lastError` / `runCount` / `failCount`. **Manual-trigger** `POST /api/cron/:name/run-now` (same admin gate) starts the job in the background and returns whether the lock was acquired. Both routes refuse a caller without the `admin` role with a plain 403 (not 404 — cron is operator infrastructure and the route's existence is documented). (7) **`NODE_ENV=test` auto-disables scheduling** (mirrors `middleware/rateLimit.js`); registrations still load and `tickOnce(name)` drives a handler synchronously, which is how the unit suite asserts handler behaviour without racing the test runner. `CRON_ENABLED=true` is the explicit override for tests that DO want the scheduler. (8) **Graceful shutdown.** `SIGTERM` / `SIGINT` listeners call `croner.stop()` on every registered job so a deploy that rolls dynos doesn't leave the in-process scheduler firing past the connection drain. (9) **Tests.** 18 unit tests via `node --test` keep the package self-contained: an in-memory mongoose stub implements `findOneAndUpdate` + `deleteOne` + the duplicate-key throw semantics with a per-collection serial queue so the "two concurrent ticks contend" assertion is deterministic; a croner stub records constructor calls so timezone / pattern flow-through is verifiable without spinning real time. Coverage spans register validation + double-register refusal, `NODE_ENV=test` auto-disable + explicit override, `tickOnce` happy-path + failure recording, two-concurrent-ticks-exactly-one-acquires, stale-lease reclamation, declarative-loader empty fallback, `lib/lock` acquire/release/heartbeat including the lost-heartbeat signal flip, admin-gated status route (non-admin 403, admin 200), run-now (success + unknown-job 404), `CRON_STATUS_PATH=''` disabling, `unregister` stopping croner + removing the job, per-job timezone, and per-job `leaseSeconds` overriding `CRON_LEASE_SECONDS`. Released via `.github/workflows/davepi-plugin-cron-publish.yml` (modelled on the queue / object-storage publish workflow because croner is a hard runtime dep) on tag pushes matching `davepi-plugin-cron@<version>`, NPM_TOKEN gated behind the `npm-publish` Environment. `release.yml` lists `davepi-plugin-cron` as a selectable target. PR #127 review pass tightened five edges before publish: (a) **Deferred index ensure now actually retries.** The previous cut logged "will retry on first tick" but had no retry path, and `lock.acquire` chained through `mongoose.connection.db.collection(...)` unguarded — a deployment where Mongo connects after `loadPlugins` would crash on the first tick and never get the unique / TTL indexes. Setup now branches on `connection.readyState`: when ready it ensures synchronously; when not, it attaches `once('connected')` + `once('open')` listeners that ensure on fire. `lib/lock.js` gained an `ensureIndexesOnce(mongoose)` helper that stashes a one-shot promise on the connection via a Symbol so `acquire()` lazily ensures on first use (covering test stubs without a `once()` API), and `acquire()` now throws a clear "connection is not ready" error instead of the cryptic property-of-undefined that a missing `connection.db` would produce. (b) **Run-now no longer misreports acquisition.** The original peek/release/reacquire pattern was racy: another node could win the lock between the peek's release and `runJob`'s re-acquire, and the HTTP response would still claim `acquired: true` while the background run silently skipped. `runJob` is now split into `acquireLease` + `executeWithLease`; run-now acquires once, returns the actual acquisition status to the HTTP caller, and the background run uses the same held lease — no second acquisition, no race window. (c) **Cooperative abort now records `status: 'aborted'`, not `'ok'`.** The README documents the pattern of returning early when `signal.aborted` becomes true; the previous status logic only flipped to `aborted` inside the `catch` branch, so a well-behaved handler that returned cleanly after a lost heartbeat would land as `ok` and `failCount` wouldn't increment — invisible in the status endpoint. After the handler returns we now check `lease.signal.aborted` and downgrade to `aborted` with a `lastError: 'lease lost mid-run'` so operators see the lease event. (d) **Heartbeats can no longer stack under DB latency.** The original `setInterval(async () => { await lease.heartbeat() })` did not await the async callback, so a slow Mongo could trigger overlapping heartbeats from a single process and distort the consecutive-failure counter. Replaced with a recursive `setTimeout` that schedules the next heartbeat ONLY after the previous one resolves — overlap is impossible by construction. New unit test instruments the Mongo stub with a 50ms delay on heartbeat queries and a 2.2s handler that would have triggered ≥2 stacked heartbeats under the old shape; the test asserts `maxInFlight ≤ 1`. (e) **Run-now error response no longer leaks `err.message`.** The router previously built a `ValidationError("could not start '${name}': ${err.message}")` that could surface Mongo / driver internals to the HTTP caller; now the caught error is logged through pino only and the response carries a generic `"could not start cron job"` — matches the framework's compliance rule for centralised error shaping. Five new tests cover the cooperative-abort path (`status === 'aborted'`, `failCount === 1`, `lastError ~ /lease lost/`), the no-overlap heartbeat under simulated DB latency, the run-now-honours-real-lock path (pre-occupied lock yields `acquired: false` and the handler never starts), the `acquire` defensive guard for an unready connection, and the deferred-ensureIndexes listener attachment when `readyState !== 1`. Suite is now 23 cases. (#115, #127)
- **`davepi-plugin-twilio@0.1.0` published.** Fifth first-party plugin under `packages/davepi-plugin-twilio/`, modelled on slack/postmark/oauth/audit/object-storage but covering Twilio's most common surfaces in a single package: outbound SMS (`sendSms`) and WhatsApp (`sendWhatsApp`) for transactional / notification sends from a schema lifecycle hook, passwordless OTP-over-SMS login (`POST /auth/otp` + `/auth/otp/verify`), TOTP-based 2FA (`POST /auth/2fa/enroll` + `/verify` + `/challenge`), and an inbound SMS webhook with `X-Twilio-Signature` verification. One package — rather than splitting transactional and auth surfaces into two — because a consumer wiring Twilio for receipt SMS already has the Twilio account, the auth-token secret, and the failure-isolation posture sorted; splitting them would duplicate config and obscure the fact that an outage on the same Twilio account propagates to every surface together. **Dormant on `TWILIO_ACCOUNT_SID` unset** so the plugin can ship in a project before Twilio is wired — `sendSms` / `sendWhatsApp` throw with a clear pointer at the missing env var, and the routes are not mounted. (1) **Outbound.** `sendSms({ to, body, messagingServiceSid?, statusCallback? })` and `sendWhatsApp({ to, templateSid?, variables?, body? })` are thin pass-throughs to the Twilio SDK's `client.messages.create` — `TWILIO_MESSAGING_SERVICE_SID` wins over `TWILIO_FROM_NUMBER` when both are set, WhatsApp prepends the `whatsapp:` prefix to `to` only if absent (so callers can pass either `+15555550100` or `whatsapp:+15555550100`), and `templateSid` switches the send to Twilio's Content API with `contentVariables: JSON.stringify(variables)` for the parameterised template path. The Twilio SDK is **lazy-required at setup time** and injection-overridable via `createPlugin({ twilioClient })` so the package's own unit suite runs without `twilio` installed. (2) **OTP login.** `POST /auth/otp` normalises the phone to E.164 via `libphonenumber-js` (with a permissive regex fallback when the dep isn't available), enforces per-phone rate limiting via a Mongo upsert on a `otp_rate` collection (atomic `$inc` inside a live 1-hour window; `OTP_MAX_ATTEMPTS_PER_HOUR` exhaustion returns 403 because `utils/errors` doesn't yet ship a `TooManyRequestsError` type — clearly-messaged 403 is the tolerable fallback), generates a cryptographically random N-digit code (`OTP_DIGITS`, default 6), stores its sha256 hash in `otp_challenge` (TTL-indexed on `expiresAt`), and dispatches the SMS body `"<APP> code: 123456"` via `sendSms`. `POST /auth/otp/verify` constant-time-compares (`crypto.timingSafeEqual`) the sha256 of the submitted code with the stored hash, increments a per-row `attempts` counter on failure (5-and-out — the row is deleted), and on success upserts a User row by `phone` with Mongoose `strict: false` (intentional: the consumer's User model evolution is consumer-owned, and `phone`/`twofaEnabled`/`totpSecretEnc`/`totpPendingEnc`/`backupCodeHashes` are not framework-declared fields). The User is then handed to the framework's `utils/tokens.issueTokenPair` so the access / refresh token shape matches every other dAvePi login. (3) **TOTP 2FA.** `enroll` generates a TOTP secret via `otplib.authenticator`, encrypts it at rest with AES-256-GCM (key derived from `TOKEN_KEY` via SHA-256, 12-byte random IV per encrypt, 16-byte auth tag — wire format `<iv>.<tag>.<ciphertext>` in hex), stores it on the user as `totpPendingEnc`, generates eight 12-char base32 backup codes drawn from a Crockford-ish alphabet (no `I` / `L` / `O` / `U` to avoid OCR mixups when a user reads a printed sheet), stores their sha256 hashes as `backupCodeHashes`, and returns the plaintext codes once at enrollment along with the `otpauth://` provisioning URL and the raw secret (for QR rendering and manual entry respectively). `verify` decrypts the pending secret, calls `authenticator.verify({ token, secret })`, and on success promotes pending → confirmed (`totpSecretEnc`) and sets `twofaEnabled: true`. `challenge` is the anonymous post-password-login surface: given `{ userId | phone, code }`, it verifies either a current TOTP **or** a one-shot backup code (removed from the hash list on use), and mints a fresh JWT via `issueTokenPair`. The plugin doesn't monkey-patch the framework's `/login` route — instead, it exposes a programmatic `verifyTotpForUser(userId, code)` helper that consumer code or a future `beforeLogin` hook can call, and the README documents the recommended SPA wire (first-factor returns `twofaRequired: true` + a challenge token, SPA POSTs the TOTP to `/auth/2fa/challenge`). (4) **Inbound webhook.** Mounted only when `TWILIO_INBOUND_PATH` is set. The handler reconstructs the full URL from `req.protocol` + `req.get('host')` + `req.originalUrl` (honouring `x-forwarded-proto` / `x-forwarded-host` when the framework has `TRUST_PROXY=true`), validates `X-Twilio-Signature` via `twilio.validateRequest(authToken, signature, fullUrl, params)`, and on mismatch delegates to `errorHandler` via `next(new UnauthorizedError(...))` so the response shape matches every other 4xx. On success, ACKs with the TwiML empty `<Response/>` (`text/xml`, the shape Twilio expects), then fans out via `setImmediate` to subscribers registered through `onInboundSms(handler)` — each call wrapped in try/catch so one bad subscriber doesn't starve the others, and a handler crash never propagates back to Twilio (Twilio retries are for transport failures, not application failures). - **`davepi-plugin-queue@0.1.0` published.** Fifth first-party plugin under `packages/davepi-plugin-queue/`, modelled on slack / postmark / audit / object-storage. The motivating gap: `after*` schema hooks run synchronously to the request (CLAUDE.md is explicit) which is fine for in-process side-effects (audit fan-out, cache invalidation) but the wrong tool the moment a hook has to send a Postmark email, hit Stripe, regenerate a PDF, or call an LLM — response time balloons and any failure surfaces as a 5xx mid-request. There was no framework primitive for "defer this to later, but durably." The plugin wraps [BullMQ](https://docs.bullmq.io/) (Redis-backed durable job queues) behind a thin `enqueue(name, data, opts)` / `registerJob(name, handler)` API and a bus channel — `bus.emit('job:enqueue', { name, data, opts })` — so a hook can defer slow work without importing the plugin module directly, preserving the loose coupling between schemas and plugins. (1) **Dormant on `QUEUE_REDIS_URL` unset** so the package is safe to depend on in a project that hasn't wired Redis yet — boot logs a warning, `enqueue()` throws with a clear message, the status route isn't mounted; matches the slack/postmark/object-storage posture. BullMQ itself is lazy-loaded inside `setup()` so dormant installs pay no module-graph cost. (2) **Tenancy is a hard invariant.** Every `enqueue()` requires either `opts.user` (typically `req.user` from a hook) or a pre-stamped `data.userId` — the plugin refuses unscoped jobs at enqueue time rather than letting them slip past the status endpoint's tenant check. `userId` (and `accountId` when present) are stamped onto `job.data` so they round-trip through Redis, and `GET /api/jobs/:id` (configurable via `QUEUE_STATUS_PATH`, empty disables) refuses to return a job whose stamped `userId` doesn't match `req.user.user_id`, returning a plain `404` rather than `403` so cross-tenant ID probing can't differentiate "doesn't exist" from "not yours" — same convention `utils/scopeResolver.js` enforces for auto-generated routes. (3) **Web / worker split** via `QUEUE_WORKER=false` on the web dyno: the same Node process can be both producer and consumer (default), or split so the web tier only enqueues and a dedicated worker dyno boots the same plugin with worker=true. Concurrency is per-worker (`QUEUE_CONCURRENCY`, default 5) and queue/prefix (`QUEUE_NAME`, `QUEUE_PREFIX`) let multiple apps share a Redis instance. (4) **Retry posture.** Defaults to `attempts: 3` with exponential backoff starting at 2s — overridable per `enqueue()` call. `QUEUE_FAILED_TTL` (accepts `7d`/`12h`/`30m`/`60s`/bare-ms) controls how long terminally-failed jobs stick around before being swept; null leaves them forever (operator inspects + manually cleans, fine for low-volume queues). (5) **Bus rebroadcast.** Worker lifecycle events (`completed`/`failed`/`stalled`) are re-emitted as `record`-shaped events on the framework bus (`job.completed`, `job.failed`, `job.stalled`) carrying `jobId` / `name` / stamped `userId` / `attempts` / `returnValue` or `error` — the audit / slack / sentry plugins can observe job lifecycle without any code change. Rule subscribers deliberately skip events whose `type` starts with `job.` so a wildcard rule won't infinite-loop on its own rebroadcast. (6) **Optional rule-based auto-enqueue** via `createPlugin({ rules: [{ events, build }] })`, the same shape `davepi-plugin-postmark` uses but with `build()` returning `{ name, data, opts }` instead of a Postmark template input — patterns support exact (`order.created`), resource-wildcard (`order.*`), and global (`*`); `build()` returning `null` opts out of an individual event so an event without the data needed (no email on the user record, say) is a clean skip not an exception. `user` defaults to the event's `userId` so tenancy stamping is automatic. Rules are validated eagerly at setup time so a typo doesn't surface only when the first event fires. (7) **Cron-style repeating jobs** are exposed at the low level by `registerJob('name', handler, { repeat: { pattern: '0 2 * * *' } })` — the plugin schedules a sentinel job under the same name with a fixed `jobId: 'repeat:<name>'` and BullMQ handles the recurrence after that. This intentionally overlaps with (and underpins) the planned `davepi-plugin-cron` (#115) which will provide the declarative `package.json` wrapper; both can coexist. (8) **Graceful shutdown.** `SIGTERM` / `SIGINT` listeners drain in-flight jobs via BullMQ's `worker.close()` + `queue.close()` so a deploy that rolls dynos doesn't lose work mid-flight. (9) **Status route registration ordering.** The router is mounted via `app.use(QUEUE_STATUS_PATH, auth(true), router)` after schemas are loaded; the framework's existing `moveErrorHandlerToEnd()` re-assertion after `loadPlugins` puts the terminal `errorHandler` after our router so the canonical `{ error: { code, message } }` shape is preserved. (10) **Failure isolation.** The `job:enqueue` bus subscriber and the worker rebroadcaster both wrap operations in try/catch and log through the framework's pino instance handed in at setup — a transient Redis blip or a misbehaving rule never crashes the request loop or surfaces as unhandledRejection. (11) **Operator footguns documented loud in the README.** "Pass plain objects not Mongoose documents" (BullMQ JSON-serialises payloads; Mongoose methods + internal state don't round-trip, the worker handler then receives `{}`), "Bulk paths intentionally do NOT invoke hooks" (the bus + queue is the documented escape hatch from CLAUDE.md), "Redis is a new infra requirement" (dormancy keeps the package safe to depend on but `enqueue()` throws when called dormant — tests inject `bullmq: { Queue, Worker }` stubs via `createPlugin` rather than spinning up real Redis). 21 unit tests via `node --test` keep the package self-contained: stub BullMQ via injected `{ Queue, Worker }` covering dormant-mode + enqueue tenancy stamping + tenancy refusal + retry-default override + worker dispatch + rebroadcast on `completed`/`failed` + the no-infinite-loop guarantee + bus-driven `job:enqueue` path + missing-name validation + rule matching + rule-build()-null skip + rule typo guard + double-`registerJob` refusal + `QUEUE_WORKER=false` worker-skip + name/prefix/concurrency flow-through + `QUEUE_FAILED_TTL` parsing + tenant-scoped status endpoint via Express (alice can read her job, bob gets 404 not 403) + `QUEUE_STATUS_PATH=''` disables the route. - **`davepi-plugin-object-storage@0.1.0` published.** Fourth first-party plugin under `packages/davepi-plugin-object-storage/`, modelled on slack/postmark/audit but covering the "client uploads straight to the bucket via presigned URLs" surface that complements — rather than replaces — the framework's in-tree `type: 'File'` field. Named "object-storage" rather than "s3" because the plugin covers four object-storage backends (AWS S3, Cloudflare R2, MinIO, Google Cloud Storage) — GCS in particular is not S3-API-compatible (different SDK, different V4 signing scheme) so an S3-suffixed package name would be misleading. Env vars stay on the `S3_*` and `GCS_*` prefixes that match the protocol / SDK they configure (most backends ARE S3-API, and operators reading these expect "object storage" semantics regardless of the package name). The in-tree field is a per-record, server-proxied multipart pipeline (multer holds the bytes in API memory, fine for avatars, broken for multi-GB uploads on serverless where Lambda's 6 MB and Vercel's 4.5 MB request-body caps make proxied upload physically impossible); the plugin is the presigned-URL pipeline (client PUTs straight to S3 / R2 / MinIO / GCS, the API server never sees the bytes, files become a first-class queryable resource rather than embedded `FileMeta`). Both posture and the README spell out which to use when. The package lists itself under `davepi.plugins` in the consumer's `package.json`, has `@aws-sdk/client-s3` + `@aws-sdk/s3-request-presigner` as runtime deps (lazy-loaded inside the AWS adapter so a gcs-only consumer pays no module-graph cost), `@google-cloud/storage` as an `optionalDependencies` (also lazy-loaded; a clear `set S3_BACKEND=aws / r2 / minio or install @google-cloud/storage` error when missing, not a cryptic MODULE_NOT_FOUND at runtime), and is **dormant on `S3_BUCKET` unset** so an operator can ship the package in a project before turning S3 on. (1) **Pluggable backend.** `lib/adapters/aws.js` covers AWS S3, Cloudflare R2, and MinIO — all three speak the S3 wire protocol so one client + one presigner handle the lot, with `S3_ENDPOINT` overriding the host and `forcePathStyle` defaulting on for MinIO (whose bare host needs bucket-in-path URLs) and off elsewhere. `lib/adapters/gcs.js` wraps `@google-cloud/storage`'s V4 signed URLs into the same `{ getSignedPutUrl, getSignedGetUrl, headObject, deleteObject, publicUrl }` interface so the routes layer doesn't know which provider it's talking to. Adapter construction failure (e.g. `S3_BACKEND=gcs` without `@google-cloud/storage` installed) propagates out of `setup()` and fails boot — the framework's `pluginLoader` contract is explicit that "Errors during plugin setup propagate — silently dropping a plugin would hide misconfiguration from operators", so the only soft-fail path the plugin allows is the documented `S3_BUCKET` unset dormancy. `plugin.adapter` exposes the live adapter for advanced consumers who need provider-specific APIs (multipart, lifecycle, listing) that the interface deliberately doesn't surface. (2) **Auto-registered `file` schema.** At `setup()` the plugin calls `schemaLoader.loadSchema({ path: 'file', collection: 'file', version: 'v1', ... })` so every standard surface (REST list / read / GraphQL / MCP / Swagger / the admin SPA) picks it up automatically — no schema file lands in the consumer's `schema/versions/` tree. `key`, `bucket`, `contentType`, `size`, `status`, `uploadedAt`, and `etag` are write-locked via the sentinel-ACL trick `davepi-plugin-audit` uses on its `audit` rows (`acl.create` / `acl.update` set to a role no real user holds, so `filterWritable` strips those keys from inbound POST / PUT bodies). `originalName` and `metadata` stay writable so a client can rename a file or attach app-specific labels through the regular REST PUT. `softDelete: false` is the deliberate choice — file records track an external mutable resource, and a tombstoned row whose blob may or may not still exist in the bucket is more confusing than helpful. `S3_FILE_PATH` overrides the schema path for consumers who already have their own `file` schema. (3) **Three plugin-owned REST routes** mounted under `S3_ROUTE_PREFIX` (default `/api/files`). `POST /upload-url` creates a `pending` record, calls the adapter for a presigned PUT URL, validates `contentType` against the optional `S3_ALLOWED_MIME` allowlist (wildcards like `image/*` honoured) and `size` against `S3_MAX_BYTES` (default 50 MiB) **at issuance time** — an attacker can't get a presigned URL for a forbidden type. `POST /:fileId/complete` HEADs the object via the adapter, refuses if the blob is absent (the client lied), refuses if the storage layer's reported `Content-Length` mismatches a client-declared `size` (the client lied at presign time and PUT something else), refuses if the actual blob exceeds `S3_MAX_BYTES` (the client lied by omitting `size`), flips `status` to `uploaded`, stamps `uploadedAt` and the storage etag, and returns the canonical row shape. `GET /:fileId/download-url` returns a short-lived (`S3_GET_URL_TTL_SECONDS`, default 600) presigned GET URL — but only after asserting `status === 'uploaded'` AND `record.userId === req.user.user_id`. `/:fileId/complete` is idempotent: a second call on an already-`uploaded` record returns the existing state without re-HEADing storage, so the client can safely retry after a flaky mobile-network 200. (4) **Tenant isolation, defence-in-depth.** Keys are generated as `<userId>/<8-hex>/<safe-original-name>` so a flat `aws s3 ls` shows ownership and a leaked key still isn't guessable for another tenant. Every route filters by `req.user.user_id`. Foreign-tenant `fileId`s return `404 NOT_FOUND` not `403 FORBIDDEN` so the response shape doesn't leak existence — same convention `utils/scopeResolver.js` enforces for the auto-generated REST routes. (5) **Cascade-delete behind an explicit env var.** The schema's `afterDelete` hook deletes the underlying storage object when `S3_CASCADE_DELETE=true`. Off by default because storage deletion is irreversible — a misconfigured admin endpoint that runs `DELETE` on every row would empty the bucket. The hook is best-effort: a storage failure during cascade is logged via the framework's pino but doesn't surface as a 5xx (matches the framework's `after*` hook contract). Defence-in-depth check before the storage delete fires: `userIdOfKey(record.key)` parses the leading segment of the stored key (always `<userId>/...` from `buildKey`) and compares it to `record.userId` — if they disagree (corrupted row, direct-Mongoose write that bypassed the write-locked `key` field, or future framework bug), the hook logs a warn and refuses to delete rather than potentially wiping another tenant's blob. (6) **Background reaper.** A `setInterval`-driven sweep (timer `unref`'d so it doesn't pin the event loop) finds `pending` records older than `S3_PUT_URL_TTL_SECONDS × S3_REAP_MULTIPLIER` (default 3× = 15 min for the default 5-min TTL), tries to delete the corresponding blob, then removes the DB row. Storage failures leave the row in place so the next sweep retries. Concurrent sweeps de-dupe via an inflight guard. `S3_REAP_ENABLED=false` disables the in-process reaper for consumers who prefer to drive cleanup from `davepi-plugin-cron` at their own cadence; `runOnce()` is exported for those callers. (7) **Programmatic API.** `createUploadUrl({ user, contentType, ... })`, `createDownloadUrl({ user, fileId })`, `deleteFile({ user, fileId })` for use inside schema lifecycle hooks or custom routes — each requires the JWT-shaped `user` parameter so tenant scoping is explicit at the call site. Download / delete return `null` / `false` for foreign-tenant files so the caller can map "not yours" to whatever response shape they want. **Upload policy (MIME allowlist + max-bytes) is enforced uniformly** through a shared `validateUploadRequest({ contentType, size, config, errors })` helper in `lib/config.js` that both the REST `POST /upload-url` route and the programmatic `createUploadUrl` call — a hook author who reaches for the JS surface instead of the HTTP route still gets the same gates, closing the bypass footgun the PR #122 review flagged. Apple has no separate userinfo endpoint — the profile lives in the `id_token` claims; `decodeIdToken` pulls `sub`, `email`, `email_verified` (which Apple sometimes ships as the *string* `"true"`; coerced). The user's name (`firstName` / `lastName`) is delivered only on the first sign-in via the form-post `user` field — the callback handler stuffs the parsed body into `extraParams` so the Apple adapter can recover it. Apple's `sub` is the stable identifier and what we persist as `providerUserId`, not the email, because Apple may return a relay address (`...@privaterelay.appleid.com`) or omit email on subsequent sign-ins. **(4) State HMAC + PKCE without server-side session storage.** Every authorize URL carries a `state` parameter that's HMAC-SHA256-signed (via `OAUTH_STATE_SECRET`, ≥ 16 chars) and carries `{ provider, nonce, ts, returnTo?, linkedUserId?, verifier? }` base64url-encoded. 10-minute TTL by default. The PKCE `code_verifier` travels in the same signed state — that's how the callback recovers it without needing Redis or a cookie. `verifyState` uses `crypto.timingSafeEqual` and rejects bad-sig / bad-payload / expired / wrong-provider tokens with `UnauthorizedError`, which flows through the framework's terminal `errorHandler` and surfaces as 401. **(5) Account linking via the `oauth_identity` collection.** New Mongoose model (`{ userId, provider, providerUserId, email, profile, linkedAt, lastLoginAt }`, unique index on `(provider, providerUserId)`) loaded lazily via `getOAuthIdentityModel()` so the package keeps mongoose as a peer-dep-only. Resolution order on each callback: (a) if the (provider, providerUserId) is already linked, use that local User (and refresh `lastLoginAt`); (b) else if a User exists with the provider-returned email, link this new identity to that user — two providers, one local user, same email; (c) else mint a new User with `OAUTH_DEFAULT_ROLES` (default `['user']`). An additional `/auth/{provider}/link` + `/link/callback` pair binds a provider to an already-authenticated user (the link mode is gated by an injected verify-auth middleware that resolves to `davepi/middleware/auth` at setup time). Link mode refuses to silently re-bind an identity that's already owned by a different user — that would be a tenant-isolation footgun. **(6) Security posture.** Four hardenings from PR #121 review: **(6a) State is authenticated-encrypted (AES-256-GCM), not just HMAC-signed.** The PKCE `code_verifier` travels in the state payload so the callback can recover it without server-side session storage; an HMAC-signed-only state is tamper-proof but NOT confidential, so the verifier would be recoverable by anyone observing the authorize URL — defeating PKCE's interception-mitigation purpose. The encryption key is derived from `OAUTH_STATE_SECRET` via SHA-256 (32 bytes for AES-256), the IV is 12 random bytes per encrypt, and the 16-byte GCM auth tag replaces the separate HMAC step (confidentiality + integrity from one primitive). `lib/state.js` exports both `sealState`/`openState` and `signState`/`verifyState` aliases — the contract (tamper-proof token carrying a payload) is unchanged from the caller's perspective; the implementation just got stronger. A unit test asserts the verifier substring is NOT recoverable from a base64-decoded state. **(6b) `returnTo` is path-only, never the redirect destination.** Previous iteration treated `?returnTo=...` as a candidate redirect URL: an attacker could initiate a real OAuth flow with `returnTo=https://evil.example/...`, trick a victim into clicking, and the tokens would land on evil.example via the success-redirect URL. Even with state HMAC (since state binding doesn't prevent the attacker from STARTING a legitimate flow with their own malicious value). Now: `safeReturnTo` only accepts strings that start with `/`, are not protocol-relative (`//foo` rejected), don't contain `://`, and are ≤ 1024 chars. Validated paths travel through the encrypted state and are appended to `OAUTH_SUCCESS_REDIRECT` as a `&returnTo=...` query parameter the SPA at the success-redirect origin can consume. The browser's redirect destination is **always** `OAUTH_SUCCESS_REDIRECT`. Sanitisation runs both at the route boundary (`req.query.returnTo`) AND inside `buildAuthorizeRedirect` (programmatic callers) for defence in depth. **(6c) Provider error responses delegate to `errorHandler`.** The earlier inline `res.status(400).json({...})` in `failureResponse` violated the framework's centralised-error-handling convention from `CLAUDE.md`. The no-failure-redirect branch now calls `next(new ValidationError(message))` so the response shape matches every other 4xx the framework emits; the redirect branch (`OAUTH_FAILURE_REDIRECT` set) stays as a 302, because a redirect is a normal response, not an error. **(6d) Apple `response_mode=form_post` callbacks are parsed.** The framework's main app only mounts `express.json()` globally, so Apple's urlencoded POST callbacks arrived with empty `req.body` and failed with `missing code`. A plugin-local `parseUrlEncodedBody` middleware (zero-dep — uses Node's built-in `URLSearchParams`) is mounted on every POST callback route in front of the shared handler; 1MB body cap matches Express's default; skips when `req.body` is already populated so a host app that mounts `express.urlencoded()` globally is honoured. Also: the provider's access token is intentionally **not** persisted by default — only the profile is needed to mint our JWT (a future "linked-token store" feature can opt in to persistence behind another env var). PKCE is on by default for every provider that supports it, even with confidential clients, because PKCE eliminates a class of code-interception attacks orthogonal to client-secret confidentiality. **(7) Tests.** 57 unit tests via `node --test` covering state round-trip + tampering + expiry + bad-secret negatives, PKCE verifier/challenge correctness, Apple JWT signed-then-verified-with-matching-public-key round-trip plus the DER→JOSE padding edge cases, each provider's authorize-URL composition + token-exchange + profile-mapping with mocked fetch (including the GitHub `/user/emails` fallback path), the find-or-create-user logic across new-user / returning-user / cross-provider-email-link / case-insensitive-email matrices, and the full plugin flow with stubbed models (dance start → callback → JWT mint, link-mode no-mint, replay-is-idempotent, state-with-wrong-secret rejected, provider-mismatch-in-state rejected). Four framework-level integration tests (`test/plugin-oauth-integration.test.js`) drive the full dance through real `pluginLoader` + real `User` model + real `oauth_identity` model + real `issueTokenPair` against `mongodb-memory-server`: end-to-end Google login → JWT → re-authentication against existing `/api/v1/...` auth-gated routes; second sign-in resolves to the same User (no duplicate); rejection of an unsigned state; Apple-style POST `application/x-www-form-urlencoded` callback parses correctly via the plugin-local parser; and the open-redirect defence (malicious `?returnTo=https://evil/...` is dropped, never survives into state). ### Fixed

- **Creating a new schema file no longer logs a scary `TypeError` while the file is still empty.** Every tutorial starts with "create `schema/versions/v1/workout.js`", but editors write the file empty first and only then do you type into it — so the hot-reload watcher's first `add` event fired while the module still exported `{}`. `schemaWatcher.js` stamped that empty object and handed it to `loader.loadSchema`, where `buildSchemaArtifacts` dereferenced `s.fields.forEach` and threw `Cannot read properties of undefined (reading 'forEach')`, which the watcher logged at ERROR level as "schema reload failed" on every single new-file creation (it recovered on the next save, but the console noise read like a real failure). Two-layer fix: (1) `schemaWatcher.js` now skips a file that doesn't yet export a usable schema — `typeof schema.path === 'string' && Array.isArray(schema.fields)` — logging at `debug` and waiting for the next save's `change` event instead of attempting a load; the editor's empty-file-then-type flow is now silent. Because `debug` is invisible at the default `info` level, a file that stays invalid across several saves (a real typo, not a transient empty file) escalates to exactly one `warn` — "schema file still not loadable after multiple saves … not registered" — via a per-file skip counter that resets on a successful load or unlink, so a genuine mistake stays discoverable without flooding the log. Regression tests in `test/hot-reload.test.js` assert an empty schema file is never handed to `loadSchema`, and that a fieldless schema rejects with a clear error rather than `Cannot read properties of undefined`.
- **`davepi-plugin-object-storage`'s GCS/S3 adapters now honour an explicit `null` SDK override instead of silently re-`require`-ing the real package, unblocking the plugin's first publish.** The package's publish workflow runs `npm install`, which installs `optionalDependencies` — so `@google-cloud/storage` *is* present on CI even though it's optional. The two "throws when the SDK is not installed" tests inject `sdkOverride: null` (and `sdkOverrides: { gcs: null }` through the plugin) to *simulate* the SDK being absent regardless of whether it's actually on the path, but `lib/adapters/gcs.js` resolved the SDK with `const sdk = sdkOverride || loadSdk()` — and `null || loadSdk()` falls through to `loadSdk()`, which `require`d the now-installed package and returned a real SDK, so the adapter never threw and the two assertions failed (2/81), failing the publish at the test stage. The fix distinguishes "no override supplied" from "explicitly null": `const sdk = sdkOverride === undefined ? loadSdk() : sdkOverride`. Production is unchanged — `lib/adapters/index.js` reaches the adapters via `createAdapter`, which passes `sdkOverride: sdkOverrides.gcs` (i.e. `undefined` when no test override), so the real package is still loaded and a genuinely-missing GCS SDK still throws the same clear "set S3_BACKEND=aws or install @google-cloud/storage" message. - **Tutorial startup log output now matches what users actually see.** The habit-tracker tutorial told readers to expect `{"level":"info","msg":"listening","port":5050}` after `npm start`, but three things made that wrong in practice: `index.js` emitted `'server listening'` rather than `'listening'`; `port` was read directly from `process.env` as a string, so production JSON would log `"port":"5050"` rather than `"port":5050`; and in development mode `utils/logger.js` activates `pino-pretty`, so output is human-readable timestamped lines rather than JSON. Root cause was in `utils/schemaWatcher.js` — the watcher tracks each file's registry key in an in-memory `fileToKey` map that is populated *only* by its own `add` / `change` handlers, but chokidar is started with `ignoreInitial: true`, so files already present at boot (the ones `app.js` loads directly and stamps with `__sourceFile`) never fire an `add` and never land in the map. - **Bumped auth/runtime dependencies, hardened JWT verification, and enabled Apollo CSRF prevention.** A Shield security scan (`/shield:shield`) surfaced nine npm-audit findings on the dependency surface, six of which were fixable without a major upgrade. (1) **`jsonwebtoken` 8.5.1 → 9.0.3.** The v8 line allows legacy / unrestricted key types into asymmetric verifiers (GHSA-8cf7-32gw-wr33 family); davepi only signs HS256 in `utils/tokens.js` so the immediate exploit path was narrow, but every login / refresh / MCP-token path rides this dep and v8 is unmaintained. v9 enforces `algorithms` on `verify()` and tightens key handling. ### Security

- **MCP `delete_{path}` now runs schema `beforeDelete` / `afterDelete` hooks, closing a governance bypass.** Delete is the one mutation with no field-level ACL to fall back on, so schemas that must refuse certain deletes do it with a `beforeDelete` hook — `skill` and `agentPersona` block agent-authored deletes, `apiClient` requires admin. MCP `create`/`update` deliberately stay hookless (the agent self-authors memory/profiles over MCP and relies on that), and bulk paths are unchanged; only the single-record delete gate is added. `test/agent-skill.test.js` adds a regression proving an `agent`-role MCP caller cannot `delete_skill` while an operator still can, and the hooks doc / CLAUDE.md / affected schema comments are updated to describe the now-covered surface.

- **Scaffolded projects ship with `#` subpath-import aliases.** `create-davepi-app` now writes an `imports` block into the generated `package.json` mapping `#plugins/*` → `./plugins/*.js`, `#lib/*` → `./lib/*.js`, `#schema/*` → `./schema/*.js`. From a schema file, plugin, or hook, `require('#plugins/postmark')` resolves to `./plugins/postmark.js` against the project root — no `../../../` ladders, no third-party `module-alias` dep. The `.js` suffix on each mapping target matters: Node's subpath-import resolver does NOT fall back to CJS extension resolution, so bare-glob targets like `"./plugins/*"` crash MODULE_NOT_FOUND. `#` (not `@`) is the right prefix because Node reserves `@`-prefixed specifiers for npm-scoped packages. - **Extensibility framework: schema lifecycle hooks + plugins.** Two officially supported extension points beyond the auto-generated CRUD surface. (1) **Per-resource lifecycle hooks** — declare a `hooks` block on a schema with any of `beforeCreate` / `afterCreate` / `beforeUpdate` / `afterUpdate` / `beforeDelete` / `afterDelete`. `before*` hooks run synchronously to the request, can mutate the persisted input (return value replaces input; `undefined` keeps it as-is), and throw to reject via the centralised `errorHandler`. `after*` hooks run after persistence and are best-effort — thrown errors are logged but never fail the response (same posture as audit and state-machine `onEnter`). Coverage: REST single-record `POST` / `PUT /:id` / `DELETE /:id` and GraphQL `{path}CreateOne` / `{path}UpdateById` / `{path}RemoveById`. Bulk paths deliberately bypass hooks — use the event bus for bulk reactions. (2) **Plugins** — module specifiers listed under the consumer project's `package.json` → `davepi.plugins` array. Each plugin exports `{ name, async setup({ app, schemaLoader, bus, log, appName }) }` and runs after every initial schema is registered, so plugins can introspect `schemaLoader.listSchemas()` and wire per-resource routes. The `bus` is the same `EventEmitter` from `utils/events.js` that fires `record` events for every CRUD mutation, so plugin event subscribers compose with the existing webhook dispatcher. After plugin setup completes, the schema loader re-asserts `errorHandler` at the tail of the middleware stack via the newly-exposed `moveErrorHandlerToEnd`. A throw during plugin `setup` fails boot deliberately — silent dropping would hide misconfiguration from operators. Removed the stale `hooks.before` stub from `schema/versions/v1/product.js` that the framework never honored.

- **Admin SPA assets returned `403 CORS_NOT_ALLOWED` for same-origin requests.** Vite emits `<script type="module" crossorigin src="/admin/assets/index-…js">` and `<link rel="stylesheet" crossorigin href="/admin/assets/index-…css">`, and the browser sends an `Origin` header for `crossorigin`-attributed elements **even on same-origin requests**. The default `CORS_ORIGINS=http://localhost:3000` doesn't include the API's own origin, so requests from `/admin/` back to `/admin/assets/*` were rejected with 403 + JSON body — manifesting as `GET .../admin/assets/...js 403 (Forbidden)` and `Refused to apply style from .../admin/assets/...css because its MIME type ('application/json') is not a supported stylesheet MIME type`. Fix: `middleware/corsConfig.js` now detects same-origin and bypasses the allowlist check, reflecting the request's Origin via a separate `cors({ origin: true })` instance. The detection parses Origin via `new URL(...)` and compares against the effective request host case-insensitively, with default-port tolerance (`example.com` matches `example.com:80` for http, `:443` for https). When `app.set('trust proxy', ...)` is enabled (via `TRUST_PROXY=true`), `X-Forwarded-Host` is honoured so reverse-proxied deployments (Caddy / nginx / PaaS LBs) match correctly; without trust proxy the header is ignored so a malicious client can't spoof it. A cross-site attacker can't trigger the bypass either: the browser sets `Host` based on the target URL, not the attacker page's origin. New `create-davepi-app@0.1.2` adds `http://localhost:${apiPort}` to the scaffolded `.env`'s `CORS_ORIGINS` as belt-and-suspenders so the admin SPA also works for users on `davepi@1.0.3` who upgrade just the scaffolder. ### Notes

- Pre-1.0 `0.x` development happened on `main` without published
  tags. v1.0.0 is the first release covered by the semver +
  deprecation policy.
- The Mongo backend is required and intentional — SQL backends
  are not supported.
- `accountId` is auto-stamped from the JWT alongside `userId`;
  custom foreign keys must be named differently (`parentAccountId`,
  `orgId`, etc.).
