### Added

- **`davepi-plugin-twilio@0.1.0` published.** Fifth first-party plugin under `packages/davepi-plugin-twilio/`, modelled on slack/postmark/oauth/audit/object-storage but covering Twilio's most common surfaces in a single package: outbound SMS (`sendSms`) and WhatsApp (`sendWhatsApp`) for transactional / notification sends from a schema lifecycle hook, passwordless OTP-over-SMS login (`POST /auth/otp` + `/auth/otp/verify`), TOTP-based 2FA (`POST /auth/2fa/enroll` + `/verify` + `/challenge`), and an inbound SMS webhook with `X-Twilio-Signature` verification. One package — rather than splitting transactional and auth surfaces into two — because a consumer wiring Twilio for receipt SMS already has the Twilio account, the auth-token secret, and the failure-isolation posture sorted; splitting them would duplicate config and obscure the fact that an outage on the same Twilio account propagates to every surface together. **Dormant on `TWILIO_ACCOUNT_SID` unset** so the plugin can ship in a project before Twilio is wired — `sendSms` / `sendWhatsApp` throw with a clear pointer at the missing env var, and the routes are not mounted. (1) **Outbound.** `sendSms({ to, body, messagingServiceSid?, statusCallback? })` and `sendWhatsApp({ to, templateSid?, variables?, body? })` are thin pass-throughs to the Twilio SDK's `client.messages.create` — `TWILIO_MESSAGING_SERVICE_SID` wins over `TWILIO_FROM_NUMBER` when both are set, WhatsApp prepends the `whatsapp:` prefix to `to` only if absent (so callers can pass either `+15555550100` or `whatsapp:+15555550100`), and `templateSid` switches the send to Twilio's Content API with `contentVariables: JSON.stringify(variables)` for the parameterised template path. The Twilio SDK is **lazy-required at setup time** and injection-overridable via `createPlugin({ twilioClient })` so the package's own unit suite runs without `twilio` installed. (2) **OTP login.** `POST /auth/otp` normalises the phone to E.164 via `libphonenumber-js` (with a permissive regex fallback when the dep isn't available), enforces per-phone rate limiting via a Mongo upsert on a `otp_rate` collection (atomic `$inc` inside a live 1-hour window; `OTP_MAX_ATTEMPTS_PER_HOUR` exhaustion returns 403 because `utils/errors` doesn't yet ship a `TooManyRequestsError` type — clearly-messaged 403 is the tolerable fallback), generates a cryptographically random N-digit code (`OTP_DIGITS`, default 6), stores its sha256 hash in `otp_challenge` (TTL-indexed on `expiresAt`), and dispatches the SMS body `"<APP> code: 123456"` via `sendSms`. `POST /auth/otp/verify` constant-time-compares (`crypto.timingSafeEqual`) the sha256 of the submitted code with the stored hash, increments a per-row `attempts` counter on failure (5-and-out — the row is deleted), and on success upserts a User row by `phone` with Mongoose `strict: false` (intentional: the consumer's User model evolution is consumer-owned, and `phone`/`twofaEnabled`/`totpSecretEnc`/`totpPendingEnc`/`backupCodeHashes` are not framework-declared fields). The User is then handed to the framework's `utils/tokens.issueTokenPair` so the access / refresh token shape matches every other dAvePi login. (3) **TOTP 2FA.** `enroll` generates a TOTP secret via `otplib.authenticator`, encrypts it at rest with AES-256-GCM (key derived from `TOKEN_KEY` via SHA-256, 12-byte random IV per encrypt, 16-byte auth tag — wire format `<iv>.<tag>.<ciphertext>` in hex), stores it on the user as `totpPendingEnc`, generates eight 12-char base32 backup codes drawn from a Crockford-ish alphabet (no `I` / `L` / `O` / `U` to avoid OCR mixups when a user reads a printed sheet), stores their sha256 hashes as `backupCodeHashes`, and returns the plaintext codes once at enrollment along with the `otpauth://` provisioning URL and the raw secret (for QR rendering and manual entry respectively). `verify` decrypts the pending secret, calls `authenticator.verify({ token, secret })`, and on success promotes pending → confirmed (`totpSecretEnc`) and sets `twofaEnabled: true`. `challenge` is the anonymous post-password-login surface: given `{ userId | phone, code }`, it verifies either a current TOTP **or** a one-shot backup code (removed from the hash list on use), and mints a fresh JWT via `issueTokenPair`. The plugin doesn't monkey-patch the framework's `/login` route — instead, it exposes a programmatic `verifyTotpForUser(userId, code)` helper that consumer code or a future `beforeLogin` hook can call, and the README documents the recommended SPA wire (first-factor returns `twofaRequired: true` + a challenge token, SPA POSTs the TOTP to `/auth/2fa/challenge`). (4) **Inbound webhook.** Mounted only when `TWILIO_INBOUND_PATH` is set. The handler reconstructs the full URL from `req.protocol` + `req.get('host')` + `req.originalUrl` (honouring `x-forwarded-proto` / `x-forwarded-host` when the framework has `TRUST_PROXY=true`), validates `X-Twilio-Signature` via `twilio.validateRequest(authToken, signature, fullUrl, params)`, and on mismatch delegates to `errorHandler` via `next(new UnauthorizedError(...))` so the response shape matches every other 4xx. On success, ACKs with the TwiML empty `<Response/>` (`text/xml`, the shape Twilio expects), then fans out via `setImmediate` to subscribers registered through `onInboundSms(handler)` — each call wrapped in try/catch so one bad subscriber doesn't starve the others, and a handler crash never propagates back to Twilio (Twilio retries are for transport failures, not application failures). The route is mounted with `express.urlencoded({ extended: false })` specifically for that path because the main app uses `express.json()` globally; a zero-dep `URLSearchParams`-based fallback ships for sandboxes without express. (5) **Tests.** 37 unit tests via `node --test` cover dormant mode, sendSms with both messaging-service and from-number variants + statusCallback pass-through, WhatsApp template + body modes + prefix handling, the OTP rate-limit / expired-code / wrong-code / attempt-counter-exhaustion / happy-path flows including JWT issuance, AES-256-GCM round-trip + tamper-detect on the auth tag + missing-key refusal, backup-code generation alphabet, 2FA enroll → verify → flip-`twofaEnabled` lifecycle, challenge with TOTP and one-shot backup code (asserting hash removal on use), and the inbound webhook's signature-mismatch / valid-200 + TwiML / fan-out / throwing-subscriber-isolation / URL-reconstruction cases. All deps (`twilio`, `otplib`, `libphonenumber-js`, `mongoose`, framework `utils/errors` / `utils/tokens` / `model/user`) are **injection-overridable via `createPlugin({ twilioClient, totp, errors, OtpChallenge, OtpRate, User, issueTokenPair, ... })`** so the package's own unit suite runs without any of them installed. Released via `.github/workflows/davepi-plugin-twilio-publish.yml` mirroring the postmark posture (tag-triggered on `davepi-plugin-twilio@<version>`, NPM_TOKEN gated behind the `npm-publish` GitHub Environment, tag-vs-package.json version check, package tests run before publish). - **`davepi-plugin-object-storage@0.1.0` published.** Fourth first-party plugin under `packages/davepi-plugin-object-storage/`, modelled on slack/postmark/audit but covering the "client uploads straight to the bucket via presigned URLs" surface that complements — rather than replaces — the framework's in-tree `type: 'File'` field. Named "object-storage" rather than "s3" because the plugin covers four object-storage backends (AWS S3, Cloudflare R2, MinIO, Google Cloud Storage) — GCS in particular is not S3-API-compatible (different SDK, different V4 signing scheme) so an S3-suffixed package name would be misleading. Env vars stay on the `S3_*` and `GCS_*` prefixes that match the protocol / SDK they configure (most backends ARE S3-API, and operators reading these expect "object storage" semantics regardless of the package name). The in-tree field is a per-record, server-proxied multipart pipeline (multer holds the bytes in API memory, fine for avatars, broken for multi-GB uploads on serverless where Lambda's 6 MB and Vercel's 4.5 MB request-body caps make proxied upload physically impossible); the plugin is the presigned-URL pipeline (client PUTs straight to S3 / R2 / MinIO / GCS, the API server never sees the bytes, files become a first-class queryable resource rather than embedded `FileMeta`). Both posture and the README spell out which to use when. The package lists itself under `davepi.plugins` in the consumer's `package.json`, has `@aws-sdk/client-s3` + `@aws-sdk/s3-request-presigner` as runtime deps (lazy-loaded inside the AWS adapter so a gcs-only consumer pays no module-graph cost), `@google-cloud/storage` as an `optionalDependencies` (also lazy-loaded; a clear `set S3_BACKEND=aws / r2 / minio or install @google-cloud/storage` error when missing, not a cryptic MODULE_NOT_FOUND at runtime), and is **dormant on `S3_BUCKET` unset** so an operator can ship the package in a project before turning S3 on. (1) **Pluggable backend.** `lib/adapters/aws.js` covers AWS S3, Cloudflare R2, and MinIO — all three speak the S3 wire protocol so one client + one presigner handle the lot, with `S3_ENDPOINT` overriding the host and `forcePathStyle` defaulting on for MinIO (whose bare host needs bucket-in-path URLs) and off elsewhere. `lib/adapters/gcs.js` wraps `@google-cloud/storage`'s V4 signed URLs into the same `{ getSignedPutUrl, getSignedGetUrl, headObject, deleteObject, publicUrl }` interface so the routes layer doesn't know which provider it's talking to. Adapter construction failure (e.g. `S3_BACKEND=gcs` without `@google-cloud/storage` installed) propagates out of `setup()` and fails boot — the framework's `pluginLoader` contract is explicit that "Errors during plugin setup propagate — silently dropping a plugin would hide misconfiguration from operators", so the only soft-fail path the plugin allows is the documented `S3_BUCKET` unset dormancy. `plugin.adapter` exposes the live adapter for advanced consumers who need provider-specific APIs (multipart, lifecycle, listing) that the interface deliberately doesn't surface. (2) **Auto-registered `file` schema.** At `setup()` the plugin calls `schemaLoader.loadSchema({ path: 'file', collection: 'file', version: 'v1', ... })` so every standard surface (REST list / read / GraphQL / MCP / Swagger / the admin SPA) picks it up automatically — no schema file lands in the consumer's `schema/versions/` tree. `key`, `bucket`, `contentType`, `size`, `status`, `uploadedAt`, and `etag` are write-locked via the sentinel-ACL trick `davepi-plugin-audit` uses on its `audit` rows (`acl.create` / `acl.update` set to a role no real user holds, so `filterWritable` strips those keys from inbound POST / PUT bodies). `originalName` and `metadata` stay writable so a client can rename a file or attach app-specific labels through the regular REST PUT. `softDelete: false` is the deliberate choice — file records track an external mutable resource, and a tombstoned row whose blob may or may not still exist in the bucket is more confusing than helpful. `S3_FILE_PATH` overrides the schema path for consumers who already have their own `file` schema. (3) **Three plugin-owned REST routes** mounted under `S3_ROUTE_PREFIX` (default `/api/files`). `POST /upload-url` creates a `pending` record, calls the adapter for a presigned PUT URL, validates `contentType` against the optional `S3_ALLOWED_MIME` allowlist (wildcards like `image/*` honoured) and `size` against `S3_MAX_BYTES` (default 50 MiB) **at issuance time** — an attacker can't get a presigned URL for a forbidden type. `POST /:fileId/complete` HEADs the object via the adapter, refuses if the blob is absent (the client lied), refuses if the storage layer's reported `Content-Length` mismatches a client-declared `size` (the client lied at presign time and PUT something else), refuses if the actual blob exceeds `S3_MAX_BYTES` (the client lied by omitting `size`), flips `status` to `uploaded`, stamps `uploadedAt` and the storage etag, and returns the canonical row shape. `GET /:fileId/download-url` returns a short-lived (`S3_GET_URL_TTL_SECONDS`, default 600) presigned GET URL — but only after asserting `status === 'uploaded'` AND `record.userId === req.user.user_id`. `/:fileId/complete` is idempotent: a second call on an already-`uploaded` record returns the existing state without re-HEADing storage, so the client can safely retry after a flaky mobile-network 200. (4) **Tenant isolation, defence-in-depth.** Keys are generated as `<userId>/<8-hex>/<safe-original-name>` so a flat `aws s3 ls` shows ownership and a leaked key still isn't guessable for another tenant. Every route filters by `req.user.user_id`. Foreign-tenant `fileId`s return `404 NOT_FOUND` not `403 FORBIDDEN` so the response shape doesn't leak existence — same convention `utils/scopeResolver.js` enforces for the auto-generated REST routes. (5) **Cascade-delete behind an explicit env var.** The schema's `afterDelete` hook deletes the underlying storage object when `S3_CASCADE_DELETE=true`. Off by default because storage deletion is irreversible — a misconfigured admin endpoint that runs `DELETE` on every row would empty the bucket. The hook is best-effort: a storage failure during cascade is logged via the framework's pino but doesn't surface as a 5xx (matches the framework's `after*` hook contract). Defence-in-depth check before the storage delete fires: `userIdOfKey(record.key)` parses the leading segment of the stored key (always `<userId>/...` from `buildKey`) and compares it to `record.userId` — if they disagree (corrupted row, direct-Mongoose write that bypassed the write-locked `key` field, or future framework bug), the hook logs a warn and refuses to delete rather than potentially wiping another tenant's blob. (6) **Background reaper.** A `setInterval`-driven sweep (timer `unref`'d so it doesn't pin the event loop) finds `pending` records older than `S3_PUT_URL_TTL_SECONDS × S3_REAP_MULTIPLIER` (default 3× = 15 min for the default 5-min TTL), tries to delete the corresponding blob, then removes the DB row. Storage failures leave the row in place so the next sweep retries. Concurrent sweeps de-dupe via an inflight guard. `S3_REAP_ENABLED=false` disables the in-process reaper for consumers who prefer to drive cleanup from `davepi-plugin-cron` at their own cadence; `runOnce()` is exported for those callers. (7) **Programmatic API.** `createUploadUrl({ user, contentType, ... })`, `createDownloadUrl({ user, fileId })`, `deleteFile({ user, fileId })` for use inside schema lifecycle hooks or custom routes — each requires the JWT-shaped `user` parameter so tenant scoping is explicit at the call site. Download / delete return `null` / `false` for foreign-tenant files so the caller can map "not yours" to whatever response shape they want. **Upload policy (MIME allowlist + max-bytes) is enforced uniformly** through a shared `validateUploadRequest({ contentType, size, config, errors })` helper in `lib/config.js` that both the REST `POST /upload-url` route and the programmatic `createUploadUrl` call — a hook author who reaches for the JS surface instead of the HTTP route still gets the same gates, closing the bypass footgun the PR #122 review flagged. (8) **Framework glue.** The plugin's setup mounts its router via `app.use(routePrefix, router)` AFTER `schemaLoader.loadSchema` already re-asserted the terminal `errorHandler` at the tail of the middleware stack — meaning the just-mounted router would sit BEHIND `errorHandler` and bypass the centralised `{ error: { code, message } }` response shape. Setup calls `schemaLoader.moveErrorHandlerToEnd()` itself after mounting; the operation is idempotent (splice + re-append) so the framework's own re-assertion in `app.js` after `loadPlugins` finishes also continues to work. (9) **Bucket CORS docs in the README.** Operators on every backend (AWS, R2, MinIO, GCS) need to allow `PUT` from the client origin; the README ships the exact JSON / mc / gsutil commands for all four so consumers don't have to assemble it from provider docs. 81 unit tests via `node --test` keep the package self-contained (config parsing + MIME-wildcard matching + the shared `validateUploadRequest` helper, key generation + slugify + collision-free 1000-run, AWS adapter with mocked SDK covering credentials / endpoint / forcePathStyle / publicUrl variants / 404-safe HEAD, GCS adapter with mocked Storage covering V4 sign / metadata fetch / 404-safe HEAD, route layer covering happy path + every validation branch + idempotent /complete + foreign-tenant 404, reaper with stale-record detection + retry-on-storage-failure + concurrent-call de-dupe, plugin-level fail-fast posture on missing schemaLoader / app / adapter-ctor failure / loadSchema throw, programmatic-API rejection of forbidden MIME + oversize + bogus size, cascade-delete on/off + the key-userId-mismatch refusal + the no-userId-prefix refusal). Six integration tests under `test/plugin-object-storage-integration.test.js` drive the framework end-to-end through `pluginLoader` + `mongodb-memory-server` + a fake in-memory adapter: the full upload-url → /complete → /download-url flow lands a row visible at `GET /api/v1/file/:id`, MIME / size allowlists are enforced at issuance, user A cannot trigger `/complete` or `/download-url` for user B's file (both return 404), and cascade-delete fires when `S3_CASCADE_DELETE=true` and the standard `DELETE /api/v1/file/:id` runs. Released via `.github/workflows/davepi-plugin-object-storage-publish.yml` mirroring the slack / postmark / audit posture (tag-triggered on `davepi-plugin-object-storage@<version>`, NPM_TOKEN gated behind the `npm-publish` GitHub Environment, tag-vs-package.json version check, package tests run before publish; the workflow `npm install`s before testing because the runtime deps on the AWS SDK matter for the test suite, mirroring the audit publish workflow). - **`matchesScopeFragment` fails closed on malformed `$and` / `$or` (PR #123 follow-up).** The in-process matcher in `utils/scopeResolver.js` is the only enforcement site for `schema.acl.scope` on GraphQL `findById` / `findByIds` (those resolvers don't expose a Mongo filter object so the predicate can't be pushed to the driver). The first pass `continue`d when `$and` / `$or` wasn't an array, which silently dropped the constraint — a misconfigured schema with `acl.scope.storefront = { $and: 'oops' }` would return every record instead of none. Both branches now return `false` on a non-array value (fail closed) and log a warning through the framework's pino instance so operators see the misconfiguration. A new test loads a schema with a deliberately malformed `$and` scope and asserts the GraphQL `findById` returns `null` for a record the storefront role would otherwise see via the `acl.list` bypass.

- **MCP read tools now apply `schema.acl.scope` (PR #123 follow-up).** REST and GraphQL enforced the mandatory per-role filter introduced with the public-read feature, but the MCP server in `utils/mcpServer.js` built its Mongo queries directly and only merged the `userId` owner-scope when the `acl.list` bypass didn't apply — leaving MCP as an alternate read surface where a role with `acl.list` bypass + `acl.scope` (e.g. storefront limited to `{ published: true }`) could call `list_<resource>`, `get_<resource>`, `history_<resource>`, `search_<resource>`, the relation list/get tools, and the file-fetch tool and get records hidden everywhere else. Imported `getRoleScopeFilter` / `applyRoleScopeFilter` from `utils/acl.js` and `$and`-ed the role-scope filter into every MCP read site (seven tools total): list, getById, history-existence check, full-text search, list-relation, get-relation, and file-URL fetch. Three new integration tests in `test/public-read.test.js` boot a real MCP server via the in-memory transport with a synthetic `storefront` user and assert: `list_public_product` returns only scope-matching records, `get_public_product` for an out-of-scope record returns `isError`, and the scope cannot be widened via the MCP `filter` argument (`{ published: false }` returns empty). Aggregation tools build their pipelines through schema-declared runners and were intentionally left to the runner author (same posture as `wrapAggregation` in GraphQL — `acl.scope` enforcement at the pipeline-construction site is the runner's responsibility, since the runner is the only thing that knows the shape of its `$match`).

- **Public-read PR review fixes (#123).** Three findings from the automated review were addressed in the same branch before merge. **(1) Privilege-escalation via `apiClient` writes.** The original schema scoped `acl.list` / `acl.delete` to admin but left create/update open — any authenticated user could `POST /api/v1/apiClient` with `{ _id: 'pk_evil', role: 'admin' }`, then send `X-Client-Id: pk_evil` and get admin-level reads across every collection. Two layers now enforce admin-only writes: field-level `acl: { create: ['admin'], update: ['admin'] }` on `_id` / `name` / `role` / `status` / `description` (so `filterWritable` strips every writable key from a non-admin body and the request fails 400 on missing required fields), plus `beforeCreate` / `beforeUpdate` / `beforeDelete` hooks that throw `ForbiddenError` when the caller is not admin (or is a client-authed caller themselves). Two new tests cover the attack: a non-admin POST returns 4xx and creates no row; a non-admin PUT against an existing storefront client cannot rewrite the role to admin. **(2) `clientAuth` middleware wrapped in `asyncHandler`.** The middleware did async Mongo work (`ApiClient.findById`) outside any wrapper, so a rejected promise from the lookup would escape Express's sync middleware contract instead of routing through the framework's centralised `errorHandler`. Now wrapped in `asyncHandler` at the factory return site. **(3) `/:id/history` route ignored role scope.** Every other REST read site applied `applyRoleScopeFilter(getRoleScopeFilter(...))` on top of the existing owner / bypass query, but the history endpoint still used the older `bypassUserScopeForList`-only predicate. A role with `acl.list` bypass and an `acl.scope` predicate (e.g. storefront limited to `{ published: true }`) could therefore call `/api/v1/<resource>/<id>/history` and get before/after/diff for out-of-scope records — an alternate read surface that leaked exactly the data `acl.scope` was supposed to hide. Fixed by `$and`-ing the role-scope filter into the existing ownership query before the existence check. Two new tests assert a storefront client cannot read history for an unpublished record (404) but can for a published one (200). **(4) Hand-written model collection name fix.** Found while writing the privilege-escalation tests: `model/apiClient.js` declared `collection: 'api_client'` while the schema-loader's auto-generated model relied on Mongoose's default pluralization to land at `api_clients`. The two were writing to different collections, so `clientAuth`'s `ApiClient.findById` couldn't see rows created through the admin REST route. Hand-written model now omits the explicit `collection` option and inherits the pluralized name, so admin CRUD and the runtime lookup hit the same docs.

- **Public read access via `X-Client-Id` and `schema.acl.scope`.** Frontends without a JWT (storefronts, marketing sites, embedded widgets) now read collections by sending an `X-Client-Id` header that resolves to a role through the existing ACL surface. The motivating gap was that every auto-generated route hard-wired `auth(true)` in `utils/schemaLoader.js`, and the ACL system only varied behaviour across roles of *authenticated* users — so exposing "published products" to an anonymous storefront required a hand-written custom route per resource, with no GraphQL parity and no audit trail of which frontend made the call. The new mechanism reuses the existing role + ACL machinery instead of introducing a parallel `/public/*` surface. **(1) `middleware/clientAuth.js` mounted on `/api` and `/graphql` ahead of `auth(true)`.** Looks up `X-Client-Id` against the new `api_client` collection and, on a match, synthesises `req.user = { user_id: <id>, roles: [<client.role>], isClient: true, clientId, clientName }`. `middleware/auth.js` was updated to early-return when `req.user.user_id` is already populated, so `auth(true)` on every schema-generated route works unchanged. Bearer wins when both headers are present (the Bearer check runs first and writes `req.user` from the JWT); the client ID stays attached as audit context for logs. A missing client ID falls through to the normal 403; an unknown or `status: 'revoked'` ID returns 401. **(2) `schema.acl.scope[role] = { ...mongoFilter }`.** New ACL slot exported from `utils/acl.js` as `getRoleScopeFilter` + `applyRoleScopeFilter`. The filter is `$and`-ed into every read for any role the caller holds, AFTER the caller's own query is parsed, so a public caller cannot widen the server-imposed predicate (`?published=false` returns nothing, not the unpublished drafts). Enforcement points: the REST list / get / file-download handlers in `utils/schemaLoader.js`, and the GraphQL `wrapFilter` / `wrapFindById` / `wrapFindByIds` resolvers in `utils/scopeResolver.js`. graphql-compose-mongoose's `findById` doesn't expose the filter object (it fetches by `_id` directly), so the scope predicate is evaluated in-process via a small `matchesScopeFragment` matcher supporting equality, `$in`, `$nin`, `$ne`, `$exists`, `$and`, and `$or` — fails closed on unsupported operators. REST list queries get the raw fragment on the Mongo driver and support the full operator surface. **(3) Writes from client-authed callers are refused outright.** REST `clientAuth` returns 403 on any non-GET, non-HEAD method (the GraphQL POST path is allowed through so queries work; mutations are blocked downstream). GraphQL `wrapCreateOne` / `wrapCreateMany` / `wrapByIdMutation` and the write branches of `wrapFilter` all call a new `refuseClientWrite(user)` that throws `ForbiddenError` when `user.isClient` is true — defence in depth so a future routing tweak can't silently expose a write surface. **(4) `apiClient` schema + `model/apiClient.js`.** Admin-only resource (`acl.list: ['admin']`, `acl.delete: ['admin']`) so client IDs are issued, named, and revoked through the standard REST / GraphQL / Swagger / admin UI / MCP surfaces. The `_id` of each row IS the public client ID — these are identifiers, not secrets, and they're baked into SPA bundles. The hand-written `model/apiClient.js` exists because `clientAuth` runs before the schema loader has finished registering auto-generated models. The two share the `api_client` collection so admin CRUD and the runtime lookup hit the same Mongo docs. **(5) SDK runtime support.** `client/davepi-runtime.ts` accepts a new `clientId?: string` option on `ApiOptions` and stamps it as `X-Client-Id` on every request via `defaultHeaders` — every generated method picks it up automatically through the shared `buildHttpClient`. No codegen changes needed; the generated client already passes `...opts` through. SPAs initialise as `createDavepiClient({ baseUrl, clientId: 'pk_storefront_live_...', getToken: () => session.accessToken })` — anonymous calls send only the client header, logged-in calls upgrade naturally to Bearer. **(6) Coverage.** New `test/public-read.test.js` exercises 14 cases: REST list returns only scope-matching records, scope cannot be widened via `?published=false`, field-level ACL strips `cost` for storefront, missing client ID 403s, unknown ID 401s, revoked ID 401s, Bearer wins when both headers present, unpublished record is 404 on get-by-id for the public role, write methods (POST/DELETE) refused, GraphQL `publicProductMany` parity. The `apiClient` admin surface is covered (admin issues an ID via REST; plain user owner-scope returns empty). The full test suite (35 suites, 480 tests) passes. **(7) Docs.** README gets an end-to-end "Public read access" section under Authentication with the curl + schema example; AGENTS.md gets a new subsection under `Authentication & Authorization` covering the role + scope + write-refusal contract and the supported `acl.scope` operator surface; CLAUDE.md gets a "Public read via client IDs" paragraph next to the existing ACL paragraph documenting the architectural shape (synthetic `req.user`, `acl.scope` enforcement points, write refusal, SDK option).

- **`davepi-plugin-audit@0.1.0` published.** Third first-party plugin under `packages/davepi-plugin-audit/`, modelled on slack/postmark but covering the "every CRUD mutation lands an immutable, queryable row" surface that any compliance / regulated-industry consumer needs. The package lists itself under `davepi.plugins` in the consumer's `package.json`, has one runtime dependency (`fast-json-patch` for the RFC 6902 `diff`), and is **dormant on `AUDIT_ENABLED=false`** so an operator can ship the package in a project before turning auditing on. (1) **Auto-registered `audit` schema.** At `setup()` the plugin calls `schemaLoader.loadSchema({ path: 'audit', collection: 'audit', version: 'v1', ... })` so every standard surface (REST, GraphQL, swagger, the `_describe` manifest, the admin SPA, MCP) picks it up automatically — no schema file lands in the consumer's `schema/versions/` tree. The schema declares `softDelete: false` (a soft-deletable audit row would still present as deletable to clients), `audit: false` (so the framework's in-tree `audit_log` pipeline doesn't double-write into a sibling collection), and `acl.list: ['admin']` (admins bypass the owner-scope filter and see cross-tenant rows; regular users see only their own actions). (2) **API-level append-only enforcement at two layers.** Every field declares `acl.create` / `acl.update` whose only allowed role is a sentinel value no real user holds — `filterWritable` strips every key from inbound POST / PUT / bulk-PUT bodies, so `$set` lands empty. The schema then declares `beforeCreate` / `beforeUpdate` / `beforeDelete` hooks that throw `ForbiddenError`, so REST single-record `POST` / `PUT /:id` / `DELETE /:id` (and their GraphQL `createOne` / `updateById` / `removeById` counterparts) return HTTP 403 with code `FORBIDDEN`. `acl.delete` is intentionally absent so admins don't get a tenant-bypass on delete either — combined with the hook above, no caller (including admins) removes a row through the standard CRUD surface. Documented gap: a direct `db.audit.deleteMany(...)` from a Mongo-shell operator, and GraphQL `auditRemoveMany` (the hook is REST-only) can still delete a caller's own rows; the README spells both out and points at the DBA-side mitigations (replica-set RBAC, S3 archive with object-lock). (3) **Subscribes to `bus.on('record', ...)`.** Every CRUD event becomes one row carrying `userId`, `accountId?`, `action`, `resource`, `resourceId`, `before`, `after`, `diff`, `filter` / `numAffected` (for bulk), `ip` / `userAgent` / `reqId` (for HTTP-scoped events), and `at`. The `diff` is an RFC 6902 JSON-Patch produced by `fast-json-patch.compare` — the reference RFC 6902 implementation, zero runtime deps of its own — so the patch round-trips through `applyPatch(before, diff)` to reconstruct `after` exactly, which the unit suite asserts directly and the integration test re-checks against the framework-emitted shape. Bulk events deliberately emit **one** row with `filter` + `numAffected` (not N rows) so a high-throughput `updateMany` doesn't explode the audit table; `AUDIT_BULK_BYPASS=true` skips bulk events entirely for ultra-hot paths. (4) **Redaction.** Before persistence, both `before` and `after` are scrubbed by `AUDIT_REDACT` (default `password,token,secret`, case-insensitive, recursive into nested objects and arrays) — the literal value is replaced with `[REDACTED]`. Independent of pino's redaction set so the audit-row scrub posture can differ from the operator-log scrub posture. (5) **TTL retention.** `AUDIT_RETENTION_DAYS` (default 365) drives a TTL index on `at`; setting `0` drops any existing TTL index so rows are kept forever. The index is reconciled at boot via `collection.createIndex` / `dropIndex` so flipping the env between deploys actually takes effect — Mongo's `expireAfterSeconds` is otherwise immutable on an existing index, so the plugin drops and recreates when the value changes. (6) **Failure isolation.** Every write is wrapped in `try/catch` and routes failures through the framework's pino instance handed in at setup; a Mongo blip never blocks the request loop or surfaces as an unhandled rejection — same posture as the slack/postmark plugins. The plugin also exports `record(entry)` for ad-hoc audit writes from a hook (e.g. a non-CRUD verb like `contract_signed`); it returns `true` / `false` rather than throwing, so a missing `try/catch` in an `after*` hook can't leak as a 5xx. **Framework changes that go with this plugin.** `utils/events.js` documents the optional `before` / `after` / `req` fields and exports a new `buildReqMeta(req)` helper that produces the narrow `{ ip, userAgent, reqId }` shape that goes onto bus events — the full Express `req` is deliberately NOT placed on the bus so subscribers can't reach into arbitrary request state. `utils/schemaLoader.js`'s REST handlers (POST, PUT/:id, soft + hard DELETE, bulk PUT, file upload, state-machine transition) now thread `req: { ip, userAgent, reqId }` (and `before` / `after` where already in scope) onto every `emitRecordEvent` call; the PUT/:id path also collapses two redundant post-update `findById` reads into a single `afterDoc` that the audit, the bus event, and the `afterUpdate` hook all share. **The legacy `record` payload key (which the in-tree webhook dispatcher serializes onto every outbound delivery) is preserved on update / soft-delete / hard-delete / transitioned events**: when `before` / `after` were introduced, the `record` field was unintentionally left undefined on those events, which would have shipped JSON `"record": undefined` to every webhook subscriber. Update events now set `record` to the projected post-update snapshot; soft-delete to the tombstoned projection; hard-delete to the pre-delete projection (the only post-state available); transitioned to the projected post-transition snapshot. All four use `projectByAcl` so a transition event's snapshot can't side-channel an ACL-restricted field (e.g. `salary`) to a webhook subscriber whose roles wouldn't see it on the regular GET path. `webhooks.test.js`'s salary-ACL scenario now actively asserts the PUT-delivered `record` is projected — locking in the regression contract. GraphQL and MCP-emitted events remain on the old shape for v1 (the plugin still writes rows for those events, just without `ip` / `userAgent` / `reqId`); threading req through `buildGraphqlContext` is a follow-up. 36 unit tests via `node --test` keep the package zero-runtime-dep (matcher, redactor, diff, TTL lifecycle, bus subscription, failure isolation, append-only hooks, `record()` helper); five integration tests under `test/plugin-audit-integration.test.js` drive the framework end-to-end through `pluginLoader` and assert (a) a REST `POST` + `PUT` + `DELETE` produces three audit rows with the correct before/after/diff shape, (b) `POST /api/v1/audit` returns 403, (c) `PUT /api/v1/audit/:id` and `DELETE /api/v1/audit/:id` against a real audit row return 403 and the row is preserved, (d) tenant scope holds for user A and the admin bypass returns cross-tenant rows, (e) `password` is `[REDACTED]` in both `before` and `after`. Released via `.github/workflows/davepi-plugin-audit-publish.yml` mirroring the slack/postmark posture (tag-triggered on `davepi-plugin-audit@<version>`, NPM_TOKEN gated behind the `npm-publish` GitHub Environment, tag-vs-package.json version check, package tests run before publish).

Link mode refuses to silently re-bind an identity that's already owned by a different user — that would be a tenant-isolation footgun. **(6) Security posture.** Four hardenings from PR #121 review: **(6a) State is authenticated-encrypted (AES-256-GCM), not just HMAC-signed.** The PKCE `code_verifier` travels in the state payload so the callback can recover it without server-side session storage; an HMAC-signed-only state is tamper-proof but NOT confidential, so the verifier would be recoverable by anyone observing the authorize URL — defeating PKCE's interception-mitigation purpose. The encryption key is derived from `OAUTH_STATE_SECRET` via SHA-256 (32 bytes for AES-256), the IV is 12 random bytes per encrypt, and the 16-byte GCM auth tag replaces the separate HMAC step (confidentiality + integrity from one primitive). `lib/state.js` exports both `sealState`/`openState` and `signState`/`verifyState` aliases — the contract (tamper-proof token carrying a payload) is unchanged from the caller's perspective; the implementation just got stronger. Even with state HMAC (since state binding doesn't prevent the attacker from STARTING a legitimate flow with their own malicious value). Now: `safeReturnTo` only accepts strings that start with `/`, are not protocol-relative (`//foo` rejected), don't contain `://`, and are ≤ 1024 chars. Validated paths travel through the encrypted state and are appended to `OAUTH_SUCCESS_REDIRECT` as a `&returnTo=...` query parameter the SPA at the success-redirect origin can consume. The browser's redirect destination is **always** `OAUTH_SUCCESS_REDIRECT`. Sanitisation runs both at the route boundary (`req.query.returnTo`) AND inside `buildAuthorizeRedirect` (programmatic callers) for defence in depth. **(6c) Provider error responses delegate to `errorHandler`.** The earlier inline `res.status(400).json({...})` in `failureResponse` violated the framework's centralised-error-handling convention from `CLAUDE.md`. The no-failure-redirect branch now calls `next(new ValidationError(message))` so the response shape matches every other 4xx the framework emits; the redirect branch (`OAUTH_FAILURE_REDIRECT` set) stays as a 302, because a redirect is a normal response, not an error. **(6d) Apple `response_mode=form_post` callbacks are parsed.** The framework's main app only mounts `express.json()` globally, so Apple's urlencoded POST callbacks arrived with empty `req.body` and failed with `missing code`. A plugin-local `parseUrlEncodedBody` middleware (zero-dep — uses Node's built-in `URLSearchParams`) is mounted on every POST callback route in front of the shared handler; 1MB body cap matches Express's default; skips when `req.body` is already populated so a host app that mounts `express.urlencoded()` globally is honoured. Also: the provider's access token is intentionally **not** persisted by default — only the profile is needed to mint our JWT (a future "linked-token store" feature can opt in to persistence behind another env var). PKCE is on by default for every provider that supports it, even with confidential clients, because PKCE eliminates a class of code-interception attacks orthogonal to client-secret confidentiality. **(7) Tests.** 57 unit tests via `node --test` covering state round-trip + tampering + expiry + bad-secret negatives, PKCE verifier/challenge correctness, Apple JWT signed-then-verified-with-matching-public-key round-trip plus the DER→JOSE padding edge cases, each provider's authorize-URL composition + token-exchange + profile-mapping with mocked fetch (including the GitHub `/user/emails` fallback path), the find-or-create-user logic across new-user / returning-user / cross-provider-email-link / case-insensitive-email matrices, and the full plugin flow with stubbed models (dance start → callback → JWT mint, link-mode no-mint, replay-is-idempotent, state-with-wrong-secret rejected, provider-mismatch-in-state rejected). ### Security

- **Bumped auth/runtime dependencies, hardened JWT verification, and enabled Apollo CSRF prevention.** A Shield security scan (`/shield:shield`) surfaced nine npm-audit findings on the dependency surface, six of which were fixable without a major upgrade. (1) **`jsonwebtoken` 8.5.1 → 9.0.3.** The v8 line allows legacy / unrestricted key types into asymmetric verifiers (GHSA-8cf7-32gw-wr33 family); davepi only signs HS256 in `utils/tokens.js` so the immediate exploit path was narrow, but every login / refresh / MCP-token path rides this dep and v8 is unmaintained. v9 enforces `algorithms` on `verify()` and tightens key handling. (2) **`jwt.verify` now pins `{ algorithms: ['HS256'] }` at all three call sites** — `app.js` (GraphQL context), `middleware/auth.js` (REST), and `bin/davepi.js` (MCP token check). Without an explicit `algorithms` allow-list, a forged token whose `alg` header is changed (the classic `alg: none` or `alg: HS256` against an RSA public key confusion) is potentially accepted; the pin closes the alg-confusion class of bug at every place a token is trusted. (3) **`express` 4.17.1 → 4.22.2.** The 4.17 line resolves into a vulnerable `qs` range (GHSA-q8mj-m7cp-5q26 — remotely triggerable DoS via `qs.stringify` crashing on null/undefined entries in comma-format arrays). Caret bump within v4; no breaking changes. v5 deliberately deferred — breaking. (4) **`nodemon` 2.0.13 → 3.1.14.** Dev-only dep that pulled vulnerable `semver` (ReDoS, GHSA-c2qf-rxjj-qqgw) via `simple-update-notifier`. Single bump kills three findings; no prod-runtime impact. (5) **`csrfPrevention: true` on the ApolloServer constructor** in `utils/schemaLoader.js`. Apollo Server v3 ships with a browser-bug XS-Search (read-only CSRF) bypass (GHSA-9q82-xgwf-vj6h) that lets a malicious page time GraphQL responses cross-origin to infer data. The CVE record is pinned to the package version and persists until the full Apollo v3 → v5 migration, but `csrfPrevention: true` neutralizes the **runtime** vector by requiring a content-type that triggers CORS preflight (or an explicit `Apollo-Require-Preflight` header) on every operation — simple-CORS GET / multipart form posts are rejected. Standard `application/json` POSTs from existing clients (including the test suite, which passed 419/420 with the same flake profile as pre-change) are unaffected. `npm audit` is now down from 9 vulns (4 high / 5 moderate) to 3 moderate, all of which are `apollo-server-core` / `apollo-server-express` / transitive `uuid@9` — gated on the separately-tracked Apollo v3 → v5 migration. Scan artifacts and the triaged risk-score report (49/100 → 91/100) under `reports/security-2026-05-27.md`.

### Security

- **Tenant ownership is enforced at every persist site.** `userId` / `accountId` were not enforced at the final pre-persist step on REST `PUT /:id`, REST bulk PUT, or post-hook on any create/update path. A client could send `{ "userId": "victim" }` in a PUT body and `$set` would rewrite ownership; a `beforeCreate` / `beforeUpdate` hook (including third-party plugin code) could return `{ ...input, userId: "attacker" }` and the framework would persist it, moving the record into another tenant's scope (since list/read filters use `{ userId: caller }`, the record would disappear from the original owner's API view and appear in the attacker's). `filterWritable` was the documented strip site but explicitly **kept** `userId` / `accountId`, relying on downstream stamping that didn't happen on every path. Fix in three layers: (1) `filterWritable` now **strips** the two protected fields instead of keeping them; (2) two new helpers exported from `utils/acl.js` — `stampTenantFields(target, user)` (create paths) and `stripTenantFields(target)` (update paths) — are called post-hook at every persist site in `utils/schemaLoader.js` and `utils/scopeResolver.js`; (3) bulk PUT additionally stamps `accountId` into `safeQuery` when the schema declares an `accountId` field (gated on declaration because Mongoose's strict mode throws on upsert when the filter references an undeclared path). `test/tenant-stamp.test.js` covers all eight attack vectors (REST POST body forge, REST POST malicious hook, REST PUT /:id body forge, REST PUT /:id malicious hook, REST bulk PUT body forge, GraphQL `createOne` input-type strip, GraphQL `createOne` malicious hook, GraphQL `updateById` malicious hook).

- **Extensibility framework: schema lifecycle hooks + plugins.** Two officially supported extension points beyond the auto-generated CRUD surface. (1) **Per-resource lifecycle hooks** — declare a `hooks` block on a schema with any of `beforeCreate` / `afterCreate` / `beforeUpdate` / `afterUpdate` / `beforeDelete` / `afterDelete`. `before*` hooks run synchronously to the request, can mutate the persisted input (return value replaces input; `undefined` keeps it as-is), and throw to reject via the centralised `errorHandler`. `after*` hooks run after persistence and are best-effort — thrown errors are logged but never fail the response (same posture as audit and state-machine `onEnter`). Coverage: REST single-record `POST` / `PUT /:id` / `DELETE /:id` and GraphQL `{path}CreateOne` / `{path}UpdateById` / `{path}RemoveById`. Bulk paths deliberately bypass hooks — use the event bus for bulk reactions. (2) **Plugins** — module specifiers listed under the consumer project's `package.json` → `davepi.plugins` array. Each plugin exports `{ name, async setup({ app, schemaLoader, bus, log, appName }) }` and runs after every initial schema is registered, so plugins can introspect `schemaLoader.listSchemas()` and wire per-resource routes. The `bus` is the same `EventEmitter` from `utils/events.js` that fires `record` events for every CRUD mutation, so plugin event subscribers compose with the existing webhook dispatcher. After plugin setup completes, the schema loader re-asserts `errorHandler` at the tail of the middleware stack via the newly-exposed `moveErrorHandlerToEnd`. A throw during plugin `setup` fails boot deliberately — silent dropping would hide misconfiguration from operators. Removed the stale `hooks.before` stub from `schema/versions/v1/product.js` that the framework never honored.

- **Admin SPA assets returned `403 CORS_NOT_ALLOWED` for same-origin requests.** Vite emits `<script type="module" crossorigin src="/admin/assets/index-…js">` and `<link rel="stylesheet" crossorigin href="/admin/assets/index-…css">`, and the browser sends an `Origin` header for `crossorigin`-attributed elements **even on same-origin requests**. The default `CORS_ORIGINS=http://localhost:3000` doesn't include the API's own origin, so requests from `/admin/` back to `/admin/assets/*` were rejected with 403 + JSON body — manifesting as `GET .../admin/assets/...js 403 (Forbidden)` and `Refused to apply style from .../admin/assets/...css because its MIME type ('application/json') is not a supported stylesheet MIME type`. Fix: `middleware/corsConfig.js` now detects same-origin and bypasses the allowlist check, reflecting the request's Origin via a separate `cors({ origin: true })` instance. The detection parses Origin via `new URL(...)` and compares against the effective request host case-insensitively, with default-port tolerance (`example.com` matches `example.com:80` for http, `:443` for https). When `app.set('trust proxy', ...)` is enabled (via `TRUST_PROXY=true`), `X-Forwarded-Host` is honoured so reverse-proxied deployments (Caddy / nginx / PaaS LBs) match correctly; without trust proxy the header is ignored so a malicious client can't spoof it. A cross-site attacker can't trigger the bypass either: the browser sets `Host` based on the target URL, not the attacker page's origin. New `create-davepi-app@0.1.2` adds `http://localhost:${apiPort}` to the scaffolded `.env`'s `CORS_ORIGINS` as belt-and-suspenders so the admin SPA also works for users on `davepi@1.0.3` who upgrade just the scaffolder.   `/graphql` in dev), CORS allowlist via `CORS_ORIGINS`, log
  redaction for `authorization`, `cookie`, `*.password`, `*.token`. (#25, #24)
- **Production posture.** `NODE_ENV=production` disables
  GraphQL playground, introspection, and verbose error
  messages (unknown errors reduce to `Internal server error`).
- **Tenant-scoped uniqueness.** Composite indexes lead with
  `userId` so `unique: true` constraints don't cross tenants.

### Notes

- Pre-1.0 `0.x` development happened on `main` without published
  tags. v1.0.0 is the first release covered by the semver +
  deprecation policy.
- The Mongo backend is required and intentional — SQL backends
  are not supported.
- `accountId` is auto-stamped from the JWT alongside `userId`;
  custom foreign keys must be named differently (`parentAccountId`,
  `orgId`, etc.).
