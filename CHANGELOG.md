## [Unreleased]

### Added

- **`@davepi/agent` learning layer, workstream B: persistent memory, conversation persistence, and the frozen-snapshot prompt (`docs/agent-learning-layer.md` §5).** Workstream A gave the agent an owned identity but it still forgot everything between requests, and the prompt prefix was reassembled every turn (no cacheable, stable prefix). This slice fixes both. **(1) Three new schema-driven resources** — `schema/versions/v1/agentMemory.js` (`agentKey`-keyed `body`, the Hermes `MEMORY.md` analog), `customerProfile.js` (`endUserKey`-keyed `preferences`/`notes`/`lastSeenAt`, the `USER.md` analog, deliberately carrying no `agentKey` so it is **shared across a tenant's agents**), and `conversation.js` (history keyed by `(userId, agentKey, channel, conversationId)` — the channel's *conversation* scope, e.g. a Slack thread, not just the end-user, so a user's separate threads/DMs never collapse into one transcript; it also stores the frozen prompt snapshot). All three declare no `acl.list`/`acl.scope` bypass, so the per-`userId` owner scope gives tenant isolation for free across REST, GraphQL, **and** MCP (covered by new tests in `test/agent-memory.test.js`, `test/customer-profile.test.js`, `test/conversation.test.js`). Unlike the operator-owned persona, memory and profiles are **self-authored**: the agent's service role writes them through the schema-generated `create_*`/`update_*` MCP tools. Conversation history is tenant data, so it lives in davepi rather than the agent host's local `store.js` (which stays auth-only — the §11.1 open question, resolved in favour of a dedicated schema). **(2) Frozen snapshot in `packages/davepi-agent/lib/promptAssembly.js` + new `lib/conversation.js`** — the assembled prefix (persona slot 1 → operating contract slot 2 → agent memory slot 4 → customer profile slot 5) is captured **once at session start**, persisted on the conversation row (`systemSnapshot`), and reused byte-for-byte every turn; a returning user past `AGENT_SESSION_IDLE_SECONDS` (default 30 min) is treated as a new session and the snapshot is re-frozen, which is how a fact recorded in session 1 surfaces in session 2 without a mid-session write ever mutating the in-flight prefix. Conversation scope comes from `channelCtx.conversationId` — the Slack adapter sets it to the thread key (`channel::thread_ts`), the HTTP channel to the logged-in user — keeping each thread a separate persisted transcript (the per-user `customerProfile` is still shared across threads). Service-mode HTTP (no `channelUserId`/`conversationId` to key on) keeps its channel-managed history round-trip but still freezes the snapshot in-process for cache stability. Memory/profile text passes through the same prompt-injection sanitizer as the persona — it matters more here because the customer profile is partly written from end-user input. **(3) Anthropic prompt caching** — `orchestrator.js` now places the frozen prefix as a `system`-role message carrying a `providerOptions.anthropic.cacheControl` breakpoint (via the Vercel AI SDK), so the long stable prefix is cached and reused within a session; cache token counts are surfaced as an info log and a `cache` event. On by default for the Anthropic provider, `LLM_PROMPT_CACHING=false` to disable. **(4) Provenance** — `updatedBy` defaults to `agent` (stamped on the hookless MCP self-authored write path) and is overridden to `operator:<id>` by `beforeCreate`/`beforeUpdate` hooks when a human edits memory/profiles via REST/GraphQL; the operating contract now also documents the live-vs-remembered boundary (slow-changing prefs → memory snapshot; order/ticket state → always a fresh tool call). New agent config knobs: `AGENT_PERSIST_CONVERSATIONS`, `AGENT_SESSION_IDLE_SECONDS`, `LLM_PROMPT_CACHING`.
- **`@davepi/agent` learning layer, workstream A: agent persona (the Hermes `SOUL.md` analog).** First slice of the agent "learning layer" RFC (`docs/agent-learning-layer.md` §4): each agent now has an owned, editable identity that leads its system prompt. The motivation is that PR #128 shipped the agent *body* (orchestration, channels, MCP-derived tools) but hard-coded one `DEFAULT_SYSTEM_PROMPT`, so a tenant running several agents (support, sales, ops) had no per-agent brand voice and no seam for the memory/skills tickets (B/C) to plug into. The fix lands as a new schema-driven resource plus a prompt-assembly seam, deliberately keeping everything tenant-isolated server-side rather than as flat files on the agent host. **(1) New `schema/versions/v1/agentPersona.js`** — fields `accountId`, `userId` (owner/operator), `agentKey` (unique per account via a `(userId, agentKey)` composite index, so one tenant hosts many agents), the four Hermes sections `identity` / `style` / `avoid` / `defaults`, a `status` enum (`active`/`archived`), and a server-written `proposedPatch`. It declares no `acl.list`/`acl.scope` bypass, so the default per-`userId` owner scope every auto-generated route enforces gives tenant isolation for free — account A's credentials cannot read account B's persona over REST, GraphQL, **or** MCP (all three covered by tests). The agent reads its persona under its own MCP identity, which in service mode owns the tenant's data; nothing here opens a cross-tenant read path. **(2) New `packages/davepi-agent/lib/promptAssembly.js`** — `orchestrator.js` now calls `assembleSystemPrompt()` instead of using the hard-coded constant. A persona renders as prompt **slot #1** ahead of the operating contract (slot #2); with no persona row, no `agentKey` configured, an archived/empty persona, or a backend that lacks the schema, it falls back to exactly the previous `DEFAULT_SYSTEM_PROMPT` so the zero-config path is byte-identical to before. A new `agent.key` config field (env `AGENT_KEY`) selects which persona this process reads. **(3) Sanitizer over persona text before it enters the prompt** — `sanitizePersonaText` strips C0 control characters, neutralises prompt-injection phrasing (`ignore previous instructions`, `new instructions:`) and forged role turns (`system:` line-starts, `<system>…</system>` tags), caps each section at 2000 chars, and logs once when it trips. This belongs in the seam from day one because later tickets feed end-user-derived text (customer profiles) through the same assembly path, making it an injection vector into a future session's identity tier. **(4) Operators own the persona; agents are write-gated on every surface.** Operators own brand voice and compliance "avoid" rules, so an `agent`-role caller must not be able to author, rewrite, or delete a live persona — and that has to hold on MCP and bulk paths, which do **not** run schema lifecycle hooks. So the gate is the two layers that *are* universal (the same posture `apiClient` uses): field-level ACL (`{ create/update: ['user','admin'] }`) on every live field — `agentKey`, `identity`, `style`, `avoid`, `defaults`, `status` — so `filterWritable` strips them from any agent-authored create/update across REST single + bulk PUT, all GraphQL create/update mutations, and MCP create/update (and because `agentKey` is `required`, an agent-authored create fails outright); plus a `beforeDelete` hook refusing agent-authored deletes on the by-id paths (delete has no field-level ACL and the agent shares the owner's `userId` in service mode). The agent's only writable field is `proposedPatch`: it proposes edits by writing that field, and an operator reviews and applies them to the live sections. (This replaces an earlier `beforeUpdate` hook that rerouted agent edits into `proposedPatch` — a hook can't enforce that on MCP/bulk, and it clobbered a pending patch on any agent update that carried no section fields.) The deployment contract is that the agent's service token carries role `['agent']` (not `user`/`admin`) while sharing the tenant `userId`, so it reads the persona it owns but cannot mutate it. Acceptance criteria — two agents on one tenant rendering distinguishably different voices from their persona rows alone, persona text sanitized before injection, the missing-persona fallback, and tenant isolation across REST/GraphQL/MCP — are each covered by tests (`test/agent-persona.test.js`, `packages/davepi-agent/test/promptAssembly.test.js`).
- **`@davepi/agent@0.1.0` published.** New first-party package under `packages/davepi-agent/`, sibling to `@davepi/mcp` but solving the opposite half of the agent story: where `@davepi/mcp` is a stdio↔HTTP byte pump that lets developer tools (Claude Desktop / Cursor / Code) connect to a dAvePi instance for editing-time use, `@davepi/agent` is an **end-user-facing chat agent** that ships pre-wired against a running dAvePi backend so any consumer project can stand up a customer-service / internal-tools chatbot with one `npx davepi-agent` invocation. The motivating gap: every davepi consumer who wanted a chatbot had to glue together an LLM SDK, an MCP client, channel adapters, an auth refresh loop, and chart/table rendering themselves — work that's substantially the same project to project because the davepi `/mcp` endpoint already exposes every schema's CRUD + relations + aggregations + audit + file ops as tools with tenant isolation, role-based ACL, and role-scoped mandatory filters enforced server-side. The agent collapses that integration to a single package. (1) **MCP client (`lib/mcpClient.js`).** Uses `@modelcontextprotocol/sdk`'s `StreamableHTTPClientTransport` against the davepi `/mcp` HTTP endpoint, NOT inlined into the davepi process the way a plugin would be — the agent runs as a separate process and authenticates as a client, which is the right posture for "an agent that acts on behalf of a specific user / role" and matches what an embedded widget or Slack bot would do. Auth headers are attached **per request**, not at client construction, so per-user deployments can rotate identity between turns without rebuilding the transport. Tool list is cached and re-pulled lazily; the SDK is dynamically imported so the package can target CommonJS while the SDK ships ESM-only modules. (2) **Two auth strategies, selectable per deployment via `AGENT_AUTH_MODE`** (`service` default, `per-user` opt-in). Service mode (`lib/auth/service.js`) reads `DAVEPI_BEARER` or `DAVEPI_CLIENT_ID` from env once — Bearer wins when both are configured, mirroring `middleware/clientAuth.js`'s posture; X-Client-Id is the right answer for an anonymous storefront widget paired with an `apiClient` row whose role has `schema.acl.scope[role]` filters declared, and davepi will refuse writes from `isClient: true` callers automatically. Per-user mode (`lib/auth/perUser.js`) stores davepi refresh tokens keyed on `(channel, channel_user_id)` in a pluggable store (default SQLite via `better-sqlite3`, `memory:` for ephemeral / test use, `STORE_URL` override) and mints fresh access tokens at `refreshSkewSeconds` (default 60s) before expiry — caching the access token so a typical turn doesn't burn a refresh round-trip. First contact from an unlinked user throws `UNLINKED` with a `linkUrl` that points at davepi's `/login?redirect_uri=<agent>/oauth/callback?nonce=<one-time>`; the agent's HTTP channel mounts the `/oauth/callback` to complete the link. The orchestrator catches `UNLINKED` and surfaces the link URL as the assistant message rather than letting it raise as a 5xx — the right UX for "you need to log in before I can look up your data." The design rule throughout: **the JWT (or X-Client-Id) IS the access boundary; the agent never re-implements ACL checks and never tries to constrain results via prompt text** ("only show user X's orders") — that's a confused-deputy bug waiting to happen, and the README states it explicitly. (3) **Orchestrator (`lib/orchestrator.js`) on the Vercel AI SDK.** One provider switch (`lib/llm/index.js`) keyed off `LLM_PROVIDER` covers Anthropic (default, `@ai-sdk/anthropic`) and OpenAI (`@ai-sdk/openai`); new providers slot in by adding one factory. MCP tool input schemas (JSON Schema, as returned by `tools/list`) flow through the AI SDK's `jsonSchema()` helper into tool definitions — no JSON-Schema-to-Zod converter shipped in the package, which would have been a tax with no payoff given the helper exists. Streaming is on by default; `streamText`'s `onStepFinish` relays tool calls and results to the channel adapter via an `onEvent` callback so each channel can decide how to render them. The system prompt explicitly tells the model NOT to add "for user X" filters and to trust empty-result tool returns instead of widening the search. (4) **Tool router (`lib/toolRouter.js`)** kicks in when the live tool count exceeds `AGENT_TOOL_LIMIT` (default 40). Most LLMs degrade past ~50 tools — too many slots in the system message, weaker tool-pick precision. Routed mode exposes only two meta-tools (`list_resources`, `use_resource(name)`) plus the render tools; once the model picks a resource, the orchestrator narrows the exposed tool list to tools whose name contains that resource for the next turn. Resource bucketing parses davepi's tool naming convention (`list_<resource>`, `get_<resource>`, `create_<resource>`, etc.) — pure client-side, no davepi change. (5) **Render tools (`lib/renderTools.js`).** Two client-synthesized tools (`render_table`, `render_chart`) the model can call when it has tabular or chartable data. Inputs are Zod-validated (columns + rows for tables; Vega-Lite v5 spec for charts) so a prompt-injected response can't smuggle raw HTML/SVG into the channel. Render events are forwarded through the orchestrator's `onEvent` stream; each channel adapter renders natively. (6) **HTTP channel (`lib/channels/http.js`).** `POST /chat` with SSE streaming (`token` / `tool_call` / `tool_result` / `render` / `final` event types) plus a non-streaming JSON envelope mode for callers without an SSE client. `GET /health` for liveness. `GET /oauth/callback` mounted only in per-user mode. CORS allowlist via `AGENT_CORS_ORIGINS`. Errors after SSE headers are surfaced as `error` events on the stream rather than HTTP status codes, since the response is mid-flight. (7) **Slack channel (`lib/channels/slack.js`) on `@slack/bolt`.** App-mention in a channel → reply in thread; DM → reply in DM. Per-conversation history kept in-memory keyed on `(channel, thread_ts)`; intentionally not persisted, restart resets — operators who need persistence can wire a store later. Render events translate channel-natively: `render_table` becomes a Block Kit `section` with a mrkdwn table (fenced code-block fallback for >10 columns), `render_chart` becomes a QuickChart image URL embedded as an image block (Vega-Lite spec → URL-encoded into the QuickChart query string, no SVG rendering inside Slack). Both HTTP and socket-mode supported via `SLACK_SOCKET_MODE`. UNLINKED errors during a Slack turn fall back to posting the link URL in-thread. (8) **Channel templates** (`lib/channels/templates/{telegram,whatsapp,widget}.js`) are checked-in stubs with the recipe for filling them in — they mirror the `http.js` / `slack.js` shape (`channelCtx` derivation, `runTurn` invocation, render-event translation) so a v2 contributor adds one file plus the relevant SDK dep without re-deriving the architecture. The embeddable widget specifically is a browser bundle, not a server-side channel — it talks to the existing `/chat` SSE endpoint. (9) **`createAgent()` / `startAgent()` programmatic API** in `index.js` for consumers who want to embed the orchestrator in their own process rather than running the `davepi-agent` bin — the same `{ config, auth, mcpClient, model }` quad the bin uses is returned, and `runTurn()` is exported so a custom channel can drive the orchestrator directly. (10) **Tests.** 38 unit tests via `node --test` keep the package self-contained: config layering (env + file + programmatic overrides), service auth (bearer vs client-id precedence, identical-headers-across-calls, link refusal), per-user auth (link nonce generation, refresh exchange against a tiny HTTP stub of davepi's `/refresh`, cached-access-token reuse within TTL, re-mint when within `refreshSkewSeconds` of expiry), tool router (CRUD-prefix bucketing, threshold trigger, `use_resource` updating router state + returning activated tools, unknown-resource error), render tools (table column normalization, chart Vega-Lite passthrough, Slack `tableToBlocks` narrow-vs-wide rendering, Slack `chartToBlocks` QuickChart URL construction), orchestrator (`adaptMcpTools` parameters shape + execute round-trip, `normalizeMcpResult` JSON-text / plain-text / error-envelope branches, UNLINKED error → assistant link-prompt path that doesn't throw out of `runTurn`). One integration test under `test/integration-mcp.test.js` boots davepi in-process via `mongodb-memory-server` (gracefully `t.skip`s if davepi isn't installed in the workspace), seeds two users, and exercises the **ACL boundary** end-to-end: each user's agent — authed as that user via service-account mode with their own JWT — sees only their own seeded `account` records through `list_account` MCP tool calls; cross-tenant rows are NOT returned. That's the safety property worth protecting; the LLM loop is just a fancy way of choosing which tool to call. (11) **Demo (`demo/server.js` + `demo/index.html`)** boots the agent with a tiny static page that opens the SSE stream and renders `render_table` as HTML and `render_chart` via `vega-embed` — used as the "did v1 actually ship" gate during development. (12) **Out of scope for v1, deliberately**: Telegram / WhatsApp / embeddable widget UI (stubs only), Gemini / Bedrock / Ollama providers, vector-memory RAG beyond the per-conversation buffer, voice, proactive agent-initiated outbound (the event bus is right there and tempting, but it's a separate product surface and would conflate "user asked the agent" with "agent decided to act"), and a fine-grained chart taxonomy — `render_chart` accepts one Vega-Lite spec, specialize later if there's demand. Peer-deps on `davepi >= 1.0.5` (optional so the agent can target a remote davepi without a local install). PR #128 review pass addressed eight findings before publish. **(a) Refresh endpoint was wrong** — the first cut hit `POST /refresh` with a non-matching payload; davepi exposes `POST /auth/refresh` with `{ refreshToken }` (confirmed at `app.js:275`), so per-user mode would have failed against a real backend. The fix updates `exchangeRefresh` and the unit-test stub. **(b) The link flow assumed a redirect-based `/login` that davepi doesn't have** — davepi's `/login` is `POST` with a JSON body, not a browser redirect. The agent now hosts the link UI itself: `GET /link/:nonce` serves a small HTML form (email + password), `POST /link/:nonce` calls davepi's `/login` server-side with those credentials, stores the resulting refresh token, and shows a success page. The refresh token never crosses the browser. **(c) Refresh tokens in URL query strings (review finding #8)** — the original `/oauth/callback` accepted `refresh_token` via `req.query`, which would have leaked the credential through HTTP logs, browser history, and Referer headers. That endpoint is now retained only as an explicit 403 refusal so legacy clients fail loudly; the form-based `/link/:nonce` flow above replaces it. **(d) HTTP per-user impersonation (review finding #9)** — the first cut trusted `channelUserId` from the `/chat` request body, which made the endpoint an impersonation primitive: any caller who knew a linked id could act as that user. In per-user mode the channel now ignores body-supplied identity entirely and reads the `channelUserId` from a signed `davepi_agent_session` cookie (HMAC-SHA256, `AGENT_SESSION_SECRET`, `HttpOnly`, `SameSite=Lax`, optional `Secure`) issued during the link flow. A caller without a valid cookie gets `401 UNLINKED` + a link URL; an absent `AGENT_SESSION_SECRET` config returns `500 CONFIG_MISSING` so the operator sees the misconfiguration loudly rather than getting silently insecure cookies. Slack continues to derive `channelUserId` from Slack's signed event payload — the platform is the trust anchor there. **(e) Tool router exposed no tools in routed mode (review finding #7)** — the original loop built `exposedMcpTools` once with `activeResource: null`, then never rebuilt the AI SDK's tool object after `use_resource` updated state, so the model would call `use_resource` successfully but find an empty MCP-tool list every turn. Reworked to a stable meta-tool pattern: `list_resources` + `use_resource` + a third `call_mcp_tool({ name, args })` that gates allowed tool names against `state.activeResource` and forwards to the MCP client when the name belongs to the chosen resource. Same AI-SDK loop, one fewer roundtrip per turn, and the tool count exposed to the model stays at ~4 regardless of how many schemas the backend has. **(f) Inline error responses + bare async handlers (review findings #1 and #2)** — the original routes used `res.status(4xx).send(...)` inline (in one place echoing `err.message`) and bare `async (req, res) => {}` handlers, both of which diverge from the framework's compliance rules and risk leaking internal details on 5xx. Added `lib/asyncHandler.js` (local to the package — matches the framework's `utils/asyncHandler.js` posture but doesn't reach across the package boundary) and `lib/errors.js` with typed `AppError` / `ValidationError` / `UnauthorizedError` / `ForbiddenError` / `NotFoundError` / `UnlinkedError`. Every HTTP route is wrapped in `asyncHandler`, throws typed errors, and a single terminal error middleware maps them to the canonical `{ error: { code, message } }` shape (and includes `linkUrl` on `UnlinkedError`). 5xx errors get a generic `"Internal error"` message in the response body and the real message stays in the pino log only. **(g) Synchronous `better-sqlite3` calls in `async` store methods (review finding #3)** — the per-user store wrapped `better-sqlite3`'s sync `get.get(...)` / `upsert.run(...)` calls inside `async` methods, blocking the event loop under load and conflating sync DB ops with an async-typed interface. Native dep dropped entirely: the per-user surface is one row per linked user with a low write rate, so an atomic JSON file is the right shape — `lib/store.js` now ships a `memory:` backend (default for tests) and a `file:` backend that serialises all mutations through a single promise chain, writes via `<path>.tmp` + atomic `rename`, and persists at `STORE_URL=file:./davepi-agent-store.json`. Legacy `sqlite:` URLs are accepted and routed transparently to the file backend so an in-flight operator doesn't have to flip env values. Three new store tests cover round-trip + delete, cross-instance persistence, and concurrent-upsert correctness. **(h) Dynamic `await import()` in CommonJS files (review finding #4)** — `lib/mcpClient.js` and `lib/orchestrator.js` use dynamic `import()` to load the Vercel AI SDK and `@modelcontextprotocol/sdk`. These packages are ESM-only and have no CommonJS entrypoint, so dynamic `import()` is the only way to load them from a CommonJS module that the rest of the davepi ecosystem requires. Replied to the comment thread; no code change because there's no `require()`-compatible path on these packages today. Test coverage now sits at 47 unit tests across nine `node --test` files (added: `http-channel.test.js` for the session-cookie / impersonation-refusal / link-flow / `/oauth/callback`-refusal behaviour, `store.test.js` for the file-backed store) plus the in-process davepi ACL-boundary integration test (gracefully skips when davepi can't be required, e.g. in a workspace where the parent project isn't installed).
- **`davepi-plugin-cron@0.1.0` published.** Sixth first-party plugin under `packages/davepi-plugin-cron/`, modelled on the slack / postmark / audit / object-storage / queue packages and covering the universal "nightly export at 2am", "reap stale uploads every 10 minutes", "send digest emails Monday 8am" need that every project eventually bolts on. The motivating gap: a naive `node-cron` install scales to one process, fine; scale to two web dynos and every job runs twice. Solving that properly — distributed lock, leader election, lease expiry, heartbeat for long runs, cooperative abort — is enough work that it belongs as one of the standard plugins. The package wraps [croner](https://github.com/Hexagon/croner) (zero-dep, timezone-aware scheduler) and adds a `cron_lock` collection with a unique index on `name` plus a TTL index on `expiresAt` so exactly one node per tick wins the upsert race and a crashed leaseholder's row gets swept by Mongo. The decision to NOT ride on `davepi-plugin-queue`'s BullMQ `repeat` (which #114 already exposes) is that this plugin uses the existing Mongo connection — zero new infra, no Redis required — and is the right answer for projects that haven't added Redis yet. Both can coexist; the README's comparison table picks a line between them. (1) **Declarative `davepi.cron` block in `package.json`** mapping `name → { schedule, handler, timezone?, leaseSeconds? }` is the primary surface. The handler spec resolves via the same posture `utils/pluginLoader.js#resolvePlugin` uses (relative paths against cwd, absolute paths as-is, package names through Node's resolution starting at cwd) and the resolved module is a function (or `{ handler, ... }` with overrides). A typo'd declaration throws at boot — a silently-disabled job is harder to diagnose than one that fails loudly. (2) **Programmatic `register(name, opts)` / `unregister(name)`** for the dynamic-list case (e.g. one job per active tenant, hot-reload during dev). `register` after `setup()` schedules immediately; double-registration of the same name throws so a misconfigured `davepi.cron` with duplicate keys doesn't silently drop one. (3) **Mongo-backed distributed lock (`lib/lock.js`).** Every tick runs `findOneAndUpdate({ name, expiresAt: { $lt: now } }, { $set: { holderId, expiresAt: now + lease } }, { upsert: true })` — the unique index on `name` makes the upsert race-safe, exactly one of N contenders' upserts succeeds, the others throw `E11000` which the plugin maps to "another holder owns it; skip this tick." The lock — not the scheduler — is the source of truth, so clock skew between dynos that causes the same cron expression to fire on two nodes within the same second is caught at the lock. A `expireAfterSeconds: 0` TTL index on `expiresAt` lets Mongo sweep rows whose owners crashed mid-run. (4) **Heartbeat + AbortSignal for long-running jobs.** While the handler runs, a background `setInterval` ticks every `leaseSeconds/3` and extends `expiresAt`. If a heartbeat returns no document (the row's `holderId` no longer matches ours because the lease elapsed and someone else took over), the lease's `AbortSignal` flips so the handler can stop cooperatively (`if (signal.aborted) return`). The timer is `unref`'d so a CLI / test that exits before the next heartbeat doesn't wait. (5) **Per-job timezone** flows from `davepi.cron[name].timezone` (or the programmatic equivalent) through to croner's `timezone` option; `CRON_DEFAULT_TZ` (default `UTC`) is the fallback for jobs that don't specify one. (6) **Admin-gated status route** `GET /api/cron` (configurable via `CRON_STATUS_PATH`, empty disables) returns the list of registered jobs with `nextRun` / `lastRun` / `lastStatus` / `lastDurationMs` / `lastError` / `runCount` / `failCount`. **Manual-trigger** `POST /api/cron/:name/run-now` (same admin gate) starts the job in the background and returns whether the lock was acquired. Both routes refuse a caller without the `admin` role with a plain 403 (not 404 — cron is operator infrastructure and the route's existence is documented). (7) **`NODE_ENV=test` auto-disables scheduling** (mirrors `middleware/rateLimit.js`); registrations still load and `tickOnce(name)` drives a handler synchronously, which is how the unit suite asserts handler behaviour without racing the test runner. `CRON_ENABLED=true` is the explicit override for tests that DO want the scheduler. (8) **Graceful shutdown.** `SIGTERM` / `SIGINT` listeners call `croner.stop()` on every registered job so a deploy that rolls dynos doesn't leave the in-process scheduler firing past the connection drain. (9) **Tests.** 18 unit tests via `node --test` keep the package self-contained: an in-memory mongoose stub implements `findOneAndUpdate` + `deleteOne` + the duplicate-key throw semantics with a per-collection serial queue so the "two concurrent ticks contend" assertion is deterministic; a croner stub records constructor calls so timezone / pattern flow-through is verifiable without spinning real time. Coverage spans register validation + double-register refusal, `NODE_ENV=test` auto-disable + explicit override, `tickOnce` happy-path + failure recording, two-concurrent-ticks-exactly-one-acquires, stale-lease reclamation, declarative-loader empty fallback, `lib/lock` acquire/release/heartbeat including the lost-heartbeat signal flip, admin-gated status route (non-admin 403, admin 200), run-now (success + unknown-job 404), `CRON_STATUS_PATH=''` disabling, `unregister` stopping croner + removing the job, per-job timezone, and per-job `leaseSeconds` overriding `CRON_LEASE_SECONDS`. Released via `.github/workflows/davepi-plugin-cron-publish.yml` (modelled on the queue / object-storage publish workflow because croner is a hard runtime dep) on tag pushes matching `davepi-plugin-cron@<version>`, NPM_TOKEN gated behind the `npm-publish` Environment. `release.yml` lists `davepi-plugin-cron` as a selectable target. PR #127 review pass tightened five edges before publish: (a) **Deferred index ensure now actually retries.** The previous cut logged "will retry on first tick" but had no retry path, and `lock.acquire` chained through `mongoose.connection.db.collection(...)` unguarded — a deployment where Mongo connects after `loadPlugins` would crash on the first tick and never get the unique / TTL indexes. - **`davepi-plugin-queue@0.1.0` published.** Fifth first-party plugin under `packages/davepi-plugin-queue/`, modelled on slack / postmark / audit / object-storage. The motivating gap: `after*` schema hooks run synchronously to the request (CLAUDE.md is explicit) which is fine for in-process side-effects (audit fan-out, cache invalidation) but the wrong tool the moment a hook has to send a Postmark email, hit Stripe, regenerate a PDF, or call an LLM — response time balloons and any failure surfaces as a 5xx mid-request. There was no framework primitive for "defer this to later, but durably." The plugin wraps [BullMQ](https://docs.bullmq.io/) (Redis-backed durable job queues) behind a thin `enqueue(name, data, opts)` / `registerJob(name, handler)` API and a bus channel — `bus.emit('job:enqueue', { name, data, opts })` — so a hook can defer slow work without importing the plugin module directly, preserving the loose coupling between schemas and plugins. (1) **Dormant on `QUEUE_REDIS_URL` unset** so the package is safe to depend on in a project that hasn't wired Redis yet — boot logs a warning, `enqueue()` throws with a clear message, the status route isn't mounted; matches the slack/postmark/object-storage posture. BullMQ itself is lazy-loaded inside `setup()` so dormant installs pay no module-graph cost. (2) **Tenancy is a hard invariant.** Every `enqueue()` requires either `opts.user` (typically `req.user` from a hook) or a pre-stamped `data.userId` — the plugin refuses unscoped jobs at enqueue time rather than letting them slip past the status endpoint's tenant check. `userId` (and `accountId` when present) are stamped onto `job.data` so they round-trip through Redis, and `GET /api/jobs/:id` (configurable via `QUEUE_STATUS_PATH`, empty disables) refuses to return a job whose stamped `userId` doesn't match `req.user.user_id`, returning a plain `404` rather than `403` so cross-tenant ID probing can't differentiate "doesn't exist" from "not yours" — same convention `utils/scopeResolver.js` enforces for auto-generated routes. (3) **Web / worker split** via `QUEUE_WORKER=false` on the web dyno: the same Node process can be both producer and consumer (default), or split so the web tier only enqueues and a dedicated worker dyno boots the same plugin with worker=true. Concurrency is per-worker (`QUEUE_CONCURRENCY`, default 5) and queue/prefix (`QUEUE_NAME`, `QUEUE_PREFIX`) let multiple apps share a Redis instance. (4) **Retry posture.** Defaults to `attempts: 3` with exponential backoff starting at 2s — overridable per `enqueue()` call. `QUEUE_FAILED_TTL` (accepts `7d`/`12h`/`30m`/`60s`/bare-ms) controls how long terminally-failed jobs stick around before being swept; null leaves them forever (operator inspects + manually cleans, fine for low-volume queues). (5) **Bus rebroadcast.** Worker lifecycle events (`completed`/`failed`/`stalled`) are re-emitted as `record`-shaped events on the framework bus (`job.completed`, `job.failed`, `job.stalled`) carrying `jobId` / `name` / stamped `userId` / `attempts` / `returnValue` or `error` — the audit / slack / sentry plugins can observe job lifecycle without any code change. Rule subscribers deliberately skip events whose `type` starts with `job.` so a wildcard rule won't infinite-loop on its own rebroadcast. (6) **Optional rule-based auto-enqueue** via `createPlugin({ rules: [{ events, build }] })`, the same shape `davepi-plugin-postmark` uses but with `build()` returning `{ name, data, opts }` instead of a Postmark template input — patterns support exact (`order.created`), resource-wildcard (`order.*`), and global (`*`); `build()` returning `null` opts out of an individual event so an event without the data needed (no email on the user record, say) is a clean skip not an exception. `user` defaults to the event's `userId` so tenancy stamping is automatic. Rules are validated eagerly at setup time so a typo doesn't surface only when the first event fires. (7) **Cron-style repeating jobs** are exposed at the low level by `registerJob('name', handler, { repeat: { pattern: '0 2 * * *' } })` — the plugin schedules a sentinel job under the same name with a fixed `jobId: 'repeat:<name>'` and BullMQ handles the recurrence after that. This intentionally overlaps with (and underpins) the planned `davepi-plugin-cron` (#115) which will provide the declarative `package.json` wrapper; both can coexist. (10) **Failure isolation.** The `job:enqueue` bus subscriber and the worker rebroadcaster both wrap operations in try/catch and log through the framework's pino instance handed in at setup — a transient Redis blip or a misbehaving rule never crashes the request loop or surfaces as unhandledRejection. ### Security

- **Bumped auth/runtime dependencies, hardened JWT verification, and enabled Apollo CSRF prevention.** A Shield security scan (`/shield:shield`) surfaced nine npm-audit findings on the dependency surface, six of which were fixable without a major upgrade. (1) **`jsonwebtoken` 8.5.1 → 9.0.3.** The v8 line allows legacy / unrestricted key types into asymmetric verifiers (GHSA-8cf7-32gw-wr33 family); davepi only signs HS256 in `utils/tokens.js` so the immediate exploit path was narrow, but every login / refresh / MCP-token path rides this dep and v8 is unmaintained. v9 enforces `algorithms` on `verify()` and tightens key handling. (2) **`jwt.verify` now pins `{ algorithms: ['HS256'] }` at all three call sites** — `app.js` (GraphQL context), `middleware/auth.js` (REST), and `bin/davepi.js` (MCP token check). Without an explicit `algorithms` allow-list, a forged token whose `alg` header is changed (the classic `alg: none` or `alg: HS256` against an RSA public key confusion) is potentially accepted; the pin closes the alg-confusion class of bug at every place a token is trusted. (3) **`express` 4.17.1 → 4.22.2.** The 4.17 line resolves into a vulnerable `qs` range (GHSA-q8mj-m7cp-5q26 — remotely triggerable DoS via `qs.stringify` crashing on null/undefined entries in comma-format arrays). Caret bump within v4; no breaking changes. v5 deliberately deferred — breaking. (4) **`nodemon` 2.0.13 → 3.1.14.** Dev-only dep that pulled vulnerable `semver` (ReDoS, GHSA-c2qf-rxjj-qqgw) via `simple-update-notifier`. Single bump kills three findings; no prod-runtime impact. (5) **`csrfPrevention: true` on the ApolloServer constructor** in `utils/schemaLoader.js`. Apollo Server v3 ships with a browser-bug XS-Search (read-only CSRF) bypass (GHSA-9q82-xgwf-vj6h) that lets a malicious page time GraphQL responses cross-origin to infer data. The CVE record is pinned to the package version and persists until the full Apollo v3 → v5 migration, but `csrfPrevention: true` neutralizes the **runtime** vector by requiring a content-type that triggers CORS preflight (or an explicit `Apollo-Require-Preflight` header) on every operation — simple-CORS GET / multipart form posts are rejected. Standard `application/json` POSTs from existing clients (including the test suite, which passed 419/420 with the same flake profile as pre-change) are unaffected. `npm audit` is now down from 9 vulns (4 high / 5 moderate) to 3 moderate, all of which are `apollo-server-core` / `apollo-server-express` / transitive `uuid@9` — gated on the separately-tracked Apollo v3 → v5 migration. Scan artifacts and the triaged risk-score report (49/100 → 91/100) under `reports/security-2026-05-27.md`.

The convention is documented in `CLAUDE.md`, `AGENTS.md`, the scaffolded `agent.md`, and the docs site under Conventions, Hooks (where a hook calls a plugin's exported helper), and Plugins.

- **Extensibility framework: schema lifecycle hooks + plugins.** Two officially supported extension points beyond the auto-generated CRUD surface. (1) **Per-resource lifecycle hooks** — declare a `hooks` block on a schema with any of `beforeCreate` / `afterCreate` / `beforeUpdate` / `afterUpdate` / `beforeDelete` / `afterDelete`. `before*` hooks run synchronously to the request, can mutate the persisted input (return value replaces input; `undefined` keeps it as-is), and throw to reject via the centralised `errorHandler`. `after*` hooks run after persistence and are best-effort — thrown errors are logged but never fail the response (same posture as audit and state-machine `onEnter`). Coverage: REST single-record `POST` / `PUT /:id` / `DELETE /:id` and GraphQL `{path}CreateOne` / `{path}UpdateById` / `{path}RemoveById`. Bulk paths deliberately bypass hooks — use the event bus for bulk reactions. (2) **Plugins** — module specifiers listed under the consumer project's `package.json` → `davepi.plugins` array. Each plugin exports `{ name, async setup({ app, schemaLoader, bus, log, appName }) }` and runs after every initial schema is registered, so plugins can introspect `schemaLoader.listSchemas()` and wire per-resource routes. The `bus` is the same `EventEmitter` from `utils/events.js` that fires `record` events for every CRUD mutation, so plugin event subscribers compose with the existing webhook dispatcher. After plugin setup completes, the schema loader re-asserts `errorHandler` at the tail of the middleware stack via the newly-exposed `moveErrorHandlerToEnd`. A throw during plugin `setup` fails boot deliberately — silent dropping would hide misconfiguration from operators. Removed the stale `hooks.before` stub from `schema/versions/v1/product.js` that the framework never honored.

- **Admin SPA assets returned `403 CORS_NOT_ALLOWED` for same-origin requests.** Vite emits `<script type="module" crossorigin src="/admin/assets/index-…js">` and `<link rel="stylesheet" crossorigin href="/admin/assets/index-…css">`, and the browser sends an `Origin` header for `crossorigin`-attributed elements **even on same-origin requests**. The default `CORS_ORIGINS=http://localhost:3000` doesn't include the API's own origin, so requests from `/admin/` back to `/admin/assets/*` were rejected with 403 + JSON body — manifesting as `GET .../admin/assets/...js 403 (Forbidden)` and `Refused to apply style from .../admin/assets/...css because its MIME type ('application/json') is not a supported stylesheet MIME type`. Fix: `middleware/corsConfig.js` now detects same-origin and bypasses the allowlist check, reflecting the request's Origin via a separate `cors({ origin: true })` instance. The detection parses Origin via `new URL(...)` and compares against the effective request host case-insensitively, with default-port tolerance (`example.com` matches `example.com:80` for http, `:443` for https). When `app.set('trust proxy', ...)` is enabled (via `TRUST_PROXY=true`), `X-Forwarded-Host` is honoured so reverse-proxied deployments (Caddy / nginx / PaaS LBs) match correctly; without trust proxy the header is ignored so a malicious client can't spoof it. A cross-site attacker can't trigger the bypass either: the browser sets `Host` based on the target URL, not the attacker page's origin. New `create-davepi-app@0.1.2` adds `http://localhost:${apiPort}` to the scaffolded `.env`'s `CORS_ORIGINS` as belt-and-suspenders so the admin SPA also works for users on `davepi@1.0.3` who upgrade just the scaffolder. (#99)

## [1.0.0] - 2026-05-10

First stable release. The schema vocabulary, REST/GraphQL/MCP
surfaces, error codes, and the `_describe` shape are now covered
by [semver](https://docs.davepi.dev/reference/stability/).

### Added

- **Schema-driven generation.** One file under `schema/versions/v1/`
  becomes a Mongoose model, REST routes, GraphQL types and resolvers,
  Swagger fragments, MCP tools, an admin SPA resource, and an entry
  in the typed-client output.
- **Tenant isolation.** `userId` (and optionally `accountId`) are
  stamped from the JWT on every write and filtered on every read.
  Non-bypassable across REST, GraphQL, MCP, aggregations, and
  relations. (#21–#52)
- **Idempotency keys.** Stripe-style `Idempotency-Key` header on
  every auto-generated `POST` and `idempotencyKey` argument on
  every `create_<path>` MCP tool. Atomic claim-execute-complete
  protocol with stable body hashing. (#49)
- **MCP server.** First-class tool surface at `POST /mcp` (HTTP)
  and `davepi mcp` (stdio). Per-schema `list_*`, `get_*`,
  `create_*`, `update_*`, `delete_*`, plus restore / history /
  search / aggregations / file lifecycle and per-relation
  navigation. (#48)
- **Capability manifest at `/_describe`.** Compact JSON describing
  every loaded schema, field, relation, aggregation, state machine,
  and surface — agent-friendly discovery in one round-trip. (#47)
- **Relations engine.** Declarative `belongsTo` / `hasOne` /
  `hasMany` accessed via `__include`, with batched queries and
  per-traversal tenant scope. (#40)
- **Aggregations.** Declarative `aggregations[]` with auto-injected
  `$match: { userId }` plus REST + GraphQL + MCP + typed-client
  surfaces. (#39)
- **Computed fields.** Read-only fields derived at response time;
  stripped from input shapes everywhere. (#50)
- **State machines.** Per-field `stateMachine` config with
  declared transitions, `onEnter` hooks, `availableTransitions`
  virtual, generated GraphQL `<path>Transition<Field>` mutation
  with enum-typed `to:` arg, and `INVALID_TRANSITION` errors. (#51)
- **Soft delete + restore + audit log.** Default-on tombstones,
  per-record audit trail with field-level diffs, retention sweeps
  (`tombstoneTtlDays`, `auditTtlDays`). (#36)
- **File fields.** `type: 'File'` with multipart upload, configurable
  `maxBytes` / `accept`, `storage` backends (`local`, `s3`, `gcs`),
  public or signed-URL visibility. (#34)
- **Full-text search.** Mark fields `searchable: true` to opt into
  the framework-managed text index, `?q=` parameter, and
  `search_<path>` MCP tool. (#35)
- **Role-based ACL.** Document-level `acl.list` / `acl.delete`
  bypass slots and field-level `acl.read` / `acl.create` /
  `acl.update`. Symmetric across REST / GraphQL / MCP / audit /
  webhooks. (#32)
- **Outbound webhooks.** HMAC-SHA256-signed deliveries on
  create / update / delete / restore / transition events, with
  exponential-backoff retries and per-attempt audit rows. (#33)
- **Hot reload.** `chokidar`-driven schema watcher rebuilds the
  full surface (REST router, GraphQL via indirection middleware,
  MCP tool list with `tools/list_changed`) in 50–150ms. Gated on
  `NODE_ENV !== 'production' && HOT_RELOAD_SCHEMAS=true`. (#31)
- **Typed TypeScript client.** `npx davepi gen-client` walks the
  schema map and emits a deterministic, fully-typed client per
  resource (relations, state machines, aggregations, file fields,
  errors). (#52)
- **Schema migrations.** `npx davepi migrate up/down/status` with
  resumable cursors and a `_davepi_migrations` table; expand-
  migrate-contract pattern. (#37)
- **Admin SPA.** Refine-based `/admin` reads `_describe` at startup
  and renders forms / tables / detail views per schema with no
  per-resource wiring. (#38)
- **`create-davepi-app` scaffolder + five templates.** `blank`,
  `crm`, `ticketing`, `content`, `b2b-saas`. Each ships with seed
  data, `.env`, `docker-compose.yml`, and `.mcp.json`. (#73)
- **Agent dev guide for downstream projects.** Canonical
  `agent.md` mirrored to `.cursorrules`, `AGENTS.md`, and
  `.claude/skills/davepi/SKILL.md` per scaffolded project. (#75)
- **Public docs site at `docs.davepi.dev`** built with Astro
  Starlight under `docs/site/`. CI auto-deploys on push to `main`. (#74, #76)

### Security
